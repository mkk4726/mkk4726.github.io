시계열 데이터를 봤을 때 뭐부터 해야하고, 기본적인 모델링이랑 이거 평가를 어떻게 하는지 모르겠네

---

> [!note]- 클로드 기본 모델링 순서 정리
> 시계열은 일반 tabular 데이터와 접근 순서가 조금 다르다. 처음에는 헷갈리기 쉬우니 큰 흐름부터 잡고 들어가면 훨씬 수월하다.
>
> 1단계 - 데이터 이해 (기본 EDA)
> - 시간 인덱스 확인: 규칙적인 간격인지(1일, 1시간, 불규칙)와 결측 타임스텝 여부 확인
> - 기본 시각화: 단순한 시계열 라인 플롯으로 패턴을 먼저 눈으로 확인
> - 스케일/단위/이상치: 급격한 스파이크, 센서 오류, 0값의 의미 점검
>
> 2단계 - 시계열 특성 파악 (시계열 전용 EDA)
> - 분해(Decomposition): Trend / Seasonality / Residual로 분해해 전체 구조 파악 (`seasonal_decompose`)
> - 정상성(Stationarity) 확인: 시각 확인 + ADF test, 비정상이면 차분(differencing) 또는 log transform
> - ACF / PACF 확인: 자기상관 구조 파악 후 AR/MA 성향 및 차수 단서 확보
>
> 3단계 - 모델링 전략 결정
> - 단순 추세 + 계절성: SARIMA, ETS, Prophet
> - 외부 변수 있음: ARIMAX, 회귀 계열
> - 비선형/복잡한 패턴: LSTM, Transformer 계열
> - 여러 변수 상호작용: VAR, 또는 ML 기반(LightGBM + lag features)
> - 무조건 딥러닝부터 시작하지 말고, 데이터가 적으면 단순 모델을 우선 고려
>
> 4단계 - Train/Validation 분리 (매우 중요)
> - 시간 순서 유지(랜덤 split 금지)
> - 뒤쪽 기간을 validation으로 분리
> - 엄밀하게는 Walk-forward validation(rolling origin) 사용
>
> 5단계 - 모델링 및 튜닝
> - 차수 결정 -> fitting -> 잔차 진단
> - 잔차가 white noise에 가까운지 확인, 패턴이 남으면 모델 재설계
>
> 6단계 - 평가 지표 선택
> - MAE: 이상치에 덜 민감, 일반적 예측
> - RMSE: 큰 오차에 큰 패널티, 큰 오차가 치명적일 때
> - MAPE: 상대 오차(%), 스케일 다른 시계열 비교
> - SMAPE: MAPE 대칭 버전, 0 근처 값이 있는 경우
>
> 흐름 요약
> 원시 데이터
> -> 시간 인덱스 정리 + 기본 시각화
> -> 분해 -> 정상성 확인 -> ACF/PACF
> -> 모델 종류 결정
> -> 시간 순 split
> -> fitting + 잔차 진단
> -> 평가 지표

EDA -> 모델링 전략 -> 모델링 -> 평가

# Seasonal Decompose 

추세 + 계절성을 먼저 분해한다.
어떤 모델이 적절할지 이 단계를 통해 확인할 수 있음


> [!note]-  seasonal decompose 기본적인 내옹
> `seasonal_decompose` 이해하기
>
> 핵심 아이디어
>
> 시계열 하나를 **3개 성분**으로 쪼개는 것:
>
> ```text
> Y(t) = Trend + Seasonality + Residual   <- 덧셈 모델 (additive)
> Y(t) = Trend x Seasonality x Residual   <- 곱셈 모델 (multiplicative)
> ```
>
> 어떤 걸 쓸지는 나중에 설명할게. 일단 코드부터.
>
> ---
>
> 기본 사용법
>
> ```python
> import pandas as pd
> import matplotlib.pyplot as plt
> from statsmodels.tsa.seasonal import seasonal_decompose
>
> # 데이터 준비 (시간 인덱스 필수!)
> df = pd.read_csv('data.csv', parse_dates=['date'], index_col='date')
> series = df['value']
>
> # 분해
> result = seasonal_decompose(
>     series,
>     model='additive',   # 또는 'multiplicative'
>     period=12           # 계절 주기: 월별이면 12, 일별이면 7 등
> )
>
> # 시각화
> result.plot()
> plt.tight_layout()
> plt.show()
> ```
>
> 결과 그래프가 4줄로 나와:
>
> ```text
> [원본]      <- 원래 데이터
> [Trend]     <- 장기 추세 (이동평균으로 추출)
> [Seasonal]  <- 반복되는 계절 패턴
> [Residual]  <- 나머지 (설명 안 된 부분)
> ```
>
> ---
>
> 각 성분에서 뭘 봐야 하나
>
> **Trend**: 전반적으로 오르는지 내리는지, 변곡점 있는지
>
> **Seasonal**: 패턴이 일정하게 반복되는지 -> 불규칙하면 `multiplicative` 고려
>
> **Residual**:
> - 랜덤하게 퍼져 있으면 -> 분해가 잘 된 것
> - 패턴이 남아 있으면 -> 모델이 뭔가를 못 잡은 것
>
> ---
>
> additive vs multiplicative 어떻게 고르나
>
> ```text
> 계절 진폭이 일정함  ->  additive
>       <-> 
> 계절 진폭이 Trend에 비례해서 커짐  ->  multiplicative
> ```
>
> 눈으로 봤을 때 데이터가 커질수록 계절 진폭도 같이 커지면 `multiplicative`.
>
> ---
>
> `period` 어떻게 설정하나
>
> | 데이터 주기 | 보통 쓰는 period |
> |---|---|
> | 월별 | 12 |
> | 일별 | 7 (요일 패턴) 또는 365 |
> | 시간별 | 24 또는 168 (7일) |
> | 분기별 | 4 |
>
> 모르면 **ACF 그래프**에서 처음으로 튀어오르는 lag를 보면 힌트가 나옴.
>
> ---
>
> 주의사항 하나
>
> `seasonal_decompose`는 내부적으로 이동평균을 써서 Trend를 추출하는데, **양 끝 데이터가 NaN으로 잘림**. 데이터가 짧으면 STL decomposition(`STL` 클래스)이 더 나은 경우가 많다.
>
> 직접 써볼 데이터가 있으면 같이 돌려보자.

>[!note]- seasonal decompose의 한계
> 날카로운 관찰이야. 맞아, 그게 `seasonal_decompose`의 구조적 한계야.
>
> 왜 그런 현상이 생기냐
>
> `seasonal_decompose`는 Trend를 **단순 이동평균**으로 추출해.
>
> ```text
> Trend = moving_average(Y, window=period)
> Seasonal = Y - Trend  (additive 기준)
> ```
>
> 문제는 이동평균이 **장기 추세만 골라내는 게 아니라**, 계절 주기보다 긴 변동까지 Trend에 넣어버린다는 점이야. 결과적으로:
>
> - **Trend** -> 추세 + 중간 주기 변동이 섞임
> - **Seasonal** -> 진짜 계절성보다 약하게 추출됨
> - **Residual** -> 설명 못 한 변동이 여기로 흘러들어옴
>
> ---
>
> 해결책: STL Decomposition
>
> 이동평균 대신 **LOESS(국소 회귀)**로 각 성분을 반복적으로 추정해서 훨씬 깔끔하게 분리돼.
>
> ```python
> from statsmodels.tsa.seasonal import STL
>
> stl = STL(series, period=7, robust=True)
> # robust=True -> 이상치 영향 줄여줌
> result = stl.fit()
> result.plot()
> plt.show()
> ```
>
> `seasonal_decompose`와 차이점:
>
> |  | seasonal_decompose | STL |
> |---|---|---|
> | Trend 추출 | 단순 이동평균 | LOESS (유연한 곡선) |
> | Seasonal | 고정 패턴 반복 | 시간에 따라 변할 수 있음 |
> | 이상치 처리 | 없음 | robust=True로 처리 가능 |
> | 양 끝 NaN | 있음 | 없음 |
>
> STL을 돌려보면 Trend와 Seasonal이 더 명확하게 분리되는 경우가 많아.

stats.seasonal_decompose -> 단순히 이동평균을 뺀걸 계절성으로 잡아서 한계가 있음.
