## 배경

안정적인 파이프라인이 어느정도 완성된 다음에는 **99% 이상의 OCR 정확도 달성**이 핵심 과제였습니다.

의료 도메인 특성상 정확도가 특히 중요했지만, 초기에는 다음과 같은 제약이 있었습니다.
- **평가용 데이터셋 자체가 부재** → 기존 모델의 성능을 객관적으로 확인 불가
- **직접 모델을 새로 개발할 수 없는 상황** → 오픈소스 모델을 활용해 목표 달성해야 함

먼저 실제 검사 이미지 **약 200건으로 테스트 데이터셋을 구축**한 뒤 평가한 결과, 기존 모델은 목표 성능에 미치지 못함을 확인했습니다. 이후 문제의 특징을 분석해 효율적인 접근법을 설계했습니다.

---

## 1. 문제의 특성 분석과 단계별 분할

OCR은 크게 2단계로 구분됩니다.
- **Text Detection** : 이미지에서 텍스트의 위치를 찾는 단계
- **Text Recognition** : 해당 위치의 텍스트를 인식하는 단계

저희 데이터는 **검사장비에서 출력되는 고정된 형식의 이미지**라는 특성이 있었습니다. 이 특성을 활용해 Text Detection 단계를 **사전 정의(룰베이스)**로 처리해 별도 모델이 필요 없도록 단순화했습니다.

이를 통해 전체 파이프라인은 **Text Recognition 단계의 정확도에만 집중**할 수 있었고, 전체 시스템 복잡도와 리소스 사용량도 크게 줄일 수 있었습니다.

---

## 2. Text Recognition 특화 모델 도입 : TrOCR

여러 모델을 탐색하던 중 **TrOCR**이라는 오픈소스 모델을 발견했습니다. 이 모델이 개발된 동기가

> "Text Detection은 추후로 미루고 Text Recognition에 특화된 모델을 먼저 만들자"

였기에, 저희 상황과 정확히 맞아떨어졌습니다.

이후 **전처리 고도화와 후처리 개선을 반복 실험**으로 검증하며 최종 성능을 끌어올렸습니다.
- 이미지 전처리 : 노이즈 제거, 대비 조정 등
- 후처리 : 도메인 사전 기반 값 보정, 형식 검증 등

결과적으로 200건으로 구성한 테스트셋에서 **99% 이상, 거의 100%에 가까운 Character-level Accuracy**를 확인할 수 있었습니다.

---

## 3. 모델 서빙 최적화 : TritonServer

<figure>
<img src="./images/OCR_tritonserver.png" alt="OCR Model" width="50%" />
<figcaption>그림1. OCR Model 배포 구조</figcaption>
</figure>

모델 배포는 **온프레미스로 진행**했고, 서빙 프레임워크로 **TritonServer**를 사용했습니다.

TritonServer를 선택한 주요 이유는 다음과 같습니다.
- **동적 배치 처리** : 요청을 묶어 처리량 극대화
- **동시성 관리** : 여러 장비에서 오는 요청을 안정적으로 처리
- 온프레미스 환경에서도 손쉽게 도입 가능

이를 통해 40~50대 검사장비에서 발생하는 요청을 안정적으로 처리할 수 있는 서빙 환경을 구축했습니다.

---

## 성과 및 배운 점

- **검증 세트(200건) 기준 Character-level Accuracy 100%** 달성
- **정확도 99%, 에러율 1% 미만**의 안정적인 파이프라인 운영
- 검사결과의 높은 정확도를 통해 의료 서비스의 신뢰성 확보

이 프로젝트를 통해 **"문제의 특성을 정확히 이해하고 불필요한 복잡도를 제거하는 것"**이 엔지니어링에서 얼마나 중요한지 깊이 체감했습니다. 일반적인 OCR 문제라면 Detection + Recognition 양쪽 모두 고도화해야 했겠지만, 우리 도메인에서는 "고정 레이아웃"이라는 특성을 활용해 Detection을 제거함으로써 훨씬 단순하고 안정적인 시스템을 만들 수 있었습니다.

또한 평가 데이터셋이 없는 상태에서 시작해 **직접 평가 체계를 구축하고 목표 성능을 달성**한 경험은, "측정할 수 없으면 개선할 수 없다"는 원칙을 실전으로 확인한 과정이었습니다.
