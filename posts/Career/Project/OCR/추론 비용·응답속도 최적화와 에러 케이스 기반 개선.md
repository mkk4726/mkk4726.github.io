## 배경

OCR 모델 정확도를 확보한 이후, 실제 운영에서 더 중요한 문제는 따로 있었습니다.
- 40~50대 검사장비에서 들어오는 요청을 안정적으로 처리할 수 있는가
- 응답속도를 사용자 체감 기준(1초 내외)으로 유지할 수 있는가
- 에러 케이스를 빠르게 분류하고 파이프라인 단위로 개선할 수 있는가

즉, 모델 성능 자체보다 **추론 비용/응답속도/운영 안정성의 균형**이 핵심이었습니다.

---

## 시스템 구성: 안정성의 단위는 "모델 1개"가 아니라 "3개 프로그램의 연결"

운영 관점에서 OCR 서비스는 아래 3개 프로그램이 동시에 안정적이어야 성립했습니다.
1. 검사실 PC에 설치되는 **이미지 전송 클라이언트 프로그램**
2. 이미지를 수신/검증/적재하는 **OCR 서버 프로그램**
3. 온프레미스 GPU에서 TrOCR를 추론하는 **모델 서빙 프로그램(Triton)**

아래 개선은 모델 품질 개선보다, 이 3개 연결점의 장애를 줄이는 데 초점을 뒀습니다.

---

## A) 이미지 전송 클라이언트: "검안사 기존 업무 흐름을 방해하지 않는 설치/연동"

### 문제

검사실 PC 환경은 제각각이고, 현장에서는 검안사가 기존 워크플로(검안 -> 결과 저장 프로그램(아이리더) 업로드)를 끊기지 않게 유지해야 했습니다. 설치나 연동이 불안정하면 서비스 도입 자체가 실패합니다.

### 해결

- 설치 절차를 단순화해 PC별 편차를 줄이고, 초기 설정 오류를 빠르게 확인할 수 있게 구성
- 아이리더 업로드 흐름을 막지 않도록 전송 타이밍/재시도 로직을 분리
- 전송 실패 시 사용자 작업을 중단시키지 않고, 백그라운드에서 복구 가능한 방식으로 처리

### 성과

- 현장 워크플로 중단 없이 OCR 연동을 적용
- 설치/초기 연동 이슈를 운영 초기에 빠르게 수습할 수 있는 구조 확보

---

## B) OCR 서버: 입력 검증·모니터링·테스트 체계로 "런타임 대참사" 예방

### 문제

운영 중에는 예상 포맷이 아닌 이미지가 들어오거나, 신규 장비 포맷이 유입되는 케이스가 반복됐습니다. 서버 수정 후 런타임 에러가 발생하면 다수 장비가 동시에 영향받아 서비스 장애로 이어질 수 있었습니다.

### 해결

- 입력 검증 레이어를 강화해 "예상 이미지/비예상 이미지"를 서버 단계에서 선분기
- 예외 케이스를 로그로 구조화해 어떤 단계(수신/전처리/후처리)에서 실패했는지 추적 가능하게 개선
- 운영 모니터링용 기능(상태 확인, 실패 패턴 집계)을 추가해 장애 조기 감지
- 수정사항 반영 시 회귀를 막기 위해 pytest 기반 테스트 코드를 확장

### 성과

- 비정상 입력으로 인한 전체 장애 전파를 줄이고 장애 범위를 국소화
- 디버깅 시간을 단축해 수정-검증-배포 사이클 안정화
- 서버 수정 시 런타임 리스크를 사전에 낮추는 개발 프로세스 정착

## C) TrOCR 온프레미스 서빙: GPU 추론 안정성과 응답 일관성 확보

### 문제

온프레미스 GPU 환경은 라이브러리/드라이버 호환 이슈와 동시 요청 부하를 동시에 관리해야 했고, 이 구간이 불안정하면 OCR 전체 서비스가 지연됩니다.

### 해결

- Triton 기반 배포로 모델 실행 환경을 표준화
- 동적 배치/동시성 설정을 조정해 순간 트래픽에서 응답 편차를 줄임
- 서버 비동기 구조와 결합해 추론 지연이 사용자 응답 경로를 직접 막지 않도록 구성

### 성과

- 평균 응답속도 5초 -> 1초 개선
- 동시 요청 상황에서도 응답 일관성을 높여 운영 안정성 강화

## 1) TrOCR Triton 서빙 최적화: "정확한 모델"을 "운영 가능한 모델"로

### 문제

초기에는 OCR 추론이 한 요청씩 순차 처리되어 동시 요청 구간에서 지연이 커졌고, 장비 수가 늘어날수록 응답시간 편차가 커졌습니다.

### 해결

TrOCR를 Triton Inference Server로 배포하고, 운영 관점에서 아래 항목을 우선 최적화했습니다.
- **동적 배치(Dynamic Batching)**: 짧은 시간 구간 요청을 묶어 GPU 처리량 향상
- **동시성 관리**: 다중 장비 요청이 겹쳐도 큐 적체가 과도하게 발생하지 않도록 조정
- **비동기 파이프라인 결합**: OCR 완료 여부와 Client 응답을 분리해 사용자 플로우 블로킹 방지

이 과정에서 목표를 "최저 latency 1회"가 아니라, **일관된 p95 응답시간** 유지에 두고 파라미터를 조정했습니다.

### 성과

- 평균 응답속도 5초 -> 1초 개선
- 동시 요청 구간에서도 서비스 체감 지연을 크게 줄임
- 모델 정확도를 유지하면서 운영 가능한 서빙 구조로 전환

---

## 2) 추론 비용 관점의 경량화: 무조건 큰 모델보다 "필요한 복잡도"

### 문제

OCR 파이프라인에서 Detection + Recognition을 모두 모델로 처리하면 정확도는 확보되더라도 운영 복잡도와 추론 비용이 빠르게 증가합니다.

### 해결

도메인 특성(고정 레이아웃)을 활용해 파이프라인을 재설계했습니다.
- Text Detection은 룰 기반으로 단순화
- 모델은 Text Recognition(TrOCR)에만 집중
- 전처리/후처리에서 해결 가능한 오류는 모델 외부에서 처리

이 방식으로 모델 호출 수와 파이프라인 복잡도를 줄여, 비용-성능 균형을 맞췄습니다.

### 성과

- 불필요한 모델 연산 제거로 추론 비용 구조 단순화
- 장애 원인 범위를 좁혀 디버깅/개선 속도 향상
- 정확도(검증셋 200건 기준 Character-level Accuracy 100%)와 운영성을 동시 확보

---

## 3) 에러 케이스 분석 기반 파이프라인 최적화

### 문제

운영 환경에서는 "모델이 틀렸다"가 아니라, 여러 유형의 문제가 섞여 나타났습니다.
- 입력 이미지 품질 저하(노이즈, 명암, 잘림)
- 장비/포맷별 필드 위치 차이
- 후처리 규칙 미정의로 인한 값 파싱 실패

이런 문제를 하나의 정확도 지표로만 보면 개선 우선순위를 정하기 어려웠습니다.

### 해결

에러를 모델 중심이 아니라 파이프라인 단계 기준으로 분류했습니다.
1. 입력/전처리 단계 이슈
2. 인식 단계 이슈
3. 후처리/검증 단계 이슈

그리고 단계별 대응을 분리해 적용했습니다.
- 전처리 파라미터 조정(노이즈 제거, 대비 보정)
- Recognition 재시도 조건 정리
- 후처리 검증 규칙 강화(형식/범위 체크, 도메인 사전 보정)

### 성과

- 에러 원인 파악 시간을 단축해 개선 루프를 빠르게 운영
- "모델 재학습"이 필요 없는 이슈를 파이프라인에서 선해결
- 운영 중 품질 저하를 국소적으로 제어할 수 있는 구조 확보

---

## 면접에서 이렇게 설명 가능

> TrOCR를 Triton으로 배포한 경험 자체보다, "어떻게 운영 지표를 기준으로 최적화했는지"가 핵심입니다.  
> 동적 배치/동시성 조정으로 응답속도를 안정화했고, 에러를 파이프라인 단계로 분해해 모델 재학습 없이 해결 가능한 영역을 먼저 줄였습니다. 결과적으로 정확도는 유지하면서 비용·속도·운영성을 함께 개선했습니다.

> 또 하나의 핵심은 모델 1개가 아니라 클라이언트-서버-서빙 3개 프로그램의 연결 안정성을 관리했다는 점입니다. 설치/연동, 입력 검증, 모니터링, 테스트 체계를 함께 설계해 운영 중 장애를 예방하고 복구 속도를 높였습니다.
