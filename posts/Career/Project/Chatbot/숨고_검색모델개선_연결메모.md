# 목적

숨고 JD의 `검색 모델 개선` 항목에 대해, 현재 챗봇 프로젝트 경험을 **어디까지는 자신 있게 말할 수 있고**, **어디서부터는 과장인지**를 구분한다. 필요한 경우 부족한 부분을 공부해서 채울 주제도 정리한다.

---

## 1. 지금 바로 말할 수 있는 것

### 안전하게 말할 수 있는 포지셔닝

- `검색 시스템 자체를 깊게 연구한 검색 엔지니어`는 아님
- 대신 `실서비스에서 검색 실패 패턴을 분석하고, retrieval 품질을 개선한 경험이 있는 데이터 사이언티스트`라고는 말할 수 있음
- 특히 `사용자 질의를 어떻게 검색 가능한 형태로 바꿀 것인가`와 `도메인 특화 retrieval failure를 어떻게 줄일 것인가` 쪽은 실제 경험으로 연결 가능

### 현재 경험 중 숨고 검색 개선과 연결 가능한 포인트

1. **검색 실패 패턴 분석**
   - 다턴 대화의 맥락 단절
   - 복합 질의 처리 실패
   - 도메인 고유명사 미인식
   - 이건 "랭킹 모델" 경험이라기보다 `query understanding`과 `retrieval failure analysis` 경험에 가까움

2. **질의 전처리 / 구조화**
   - Query Rewriting
   - Query Decomposition
   - Keyword Extraction
   - 숨고 맥락으로 옮기면 `사용자 검색 의도를 검색 가능한 표현으로 변환한 경험`이라고 정리 가능

3. **하이브리드 검색 감각**
   - semantic search만으로는 안 되는 케이스를 키워드 필터링으로 보완
   - 숨고 맥락으로 옮기면 `시멘틱 유사도만으로 해결되지 않는 검색 실패를 룰/키워드/메타데이터와 결합해 보완한 경험`

4. **임베딩 평가와 선택**
   - 한국어 FAQ 데이터셋에 맞는 임베딩 모델 비교 평가
   - 검색 품질을 모델 선택의 문제로만 보지 않고 평가셋 구축까지 포함해서 접근

5. **검색 품질을 운영 관점에서 봄**
   - 만족/불만족 피드백 수집
   - 모니터링 자동화
   - 단순 오프라인 retrieval score보다 실제 사용자 반응과 운영 효율까지 같이 봤다는 점

---

## 2. 조심해야 하는 것

### 지금 상태에서 과장에 가까운 표현

- `검색 랭킹 모델을 개발했다`
- `검색 엔진을 고도화했다`라고만 쓰면서 내부가 전부 retrieval pipeline 개선인 것을 숨기는 표현
- `추천/매칭/검색 전반을 다 했다`
- `BM25, learning to rank, 검색 로그 기반 랭킹 최적화`를 직접 한 것처럼 보이게 쓰는 표현
- `대규모 검색 인프라 운영 경험`이 있는 것처럼 쓰는 표현

### 왜 조심해야 하나

내 현재 기록은 분명히 `retrieval 품질 개선` 경험은 있는데, 숨고 JD의 문장을 엄밀히 읽으면 여기엔 다음 가능성이 다 들어 있다.

- query understanding
- retrieval
- ranking
- 검색 로그 분석
- 온라인 실험
- 대규모 서비스 검색 최적화

그런데 내 경험은 이 중에서 **retrieval 개선 + query understanding + 운영 모니터링** 쪽이 중심이다. `ranking model`, `search log 기반 relevance optimization`, `LTR` 같은 표현은 현재 근거가 약하다.

## 3. 그래서 어떻게 포장하는 게 맞는가

### 추천 문장 1

`챗봇 RAG 환경에서 검색 실패 패턴을 분석하고, 질의 재정의·질문 분해·키워드 추출·임베딩 평가를 통해 retrieval 품질을 개선한 경험이 있습니다.`

### 추천 문장 2

`전통적인 검색 랭킹 모델 경험은 제한적이지만, 실제 서비스에서 사용자의 질문을 검색 가능한 구조로 바꾸고, 도메인 특화 검색 실패를 줄이는 문제를 직접 다뤄봤습니다.`

### 추천 문장 3

`검색을 모델 하나의 문제가 아니라 질의 구조화, 검색 전략, 평가셋 구축, 운영 모니터링까지 포함한 시스템 문제로 다뤄본 경험이 있습니다.`

### 숨고 이력서용으로 바꾸면

- 강하게: `RAG 기반 서비스에서 query understanding과 retrieval 품질 개선을 주도`
- 중간: `검색 실패 패턴 분석과 retrieval pipeline 개선 경험 보유`
- 약하게: `검색 모델 개선과 인접한 retrieval 개선 경험 보유`

지금 상태에서는 첫 번째와 두 번째 사이가 가장 안전해 보임.

## 4. 내가 실제로 했는지 더 확인이 필요한 것

아래 항목은 나중에 대화로 채워야 한다. 실제로 했으면 강한 근거가 되고, 안 했으면 빼야 한다.

### 확인 질문

1. 검색 실패 사례를 로그로 얼마나 봤나?
   - 직접 대화 로그를 읽으며 failure taxonomy를 만들었는지
   - 불만족 피드백과 연결했는지

2. Qdrant에서 실제 사용한 검색 옵션은 뭐였나?
   - metadata filter는 확인됨
   - hybrid search를 진짜로 썼는지, 아니면 semantic + metadata filtering인지 더 명확히 할 필요 있음
   - collection 분리
   - reranking 유무

3. 재정의 / 쪼개기 / 키워드 추출 중 실제 임팩트가 가장 컸던 것은 무엇이었나?
   - 면접에서는 모든 걸 다 말하기보다 가장 임팩트 큰 1~2개를 중심으로 말하는 게 좋음

---

## 4-1. 지금 확인된 근거

### Hit Rate 측정 방식

- `테스트셋을 직접 구축해서 측정했다`는 근거가 생김
- 구축 방식:
  - 제공된 문서를 바탕으로
  - LLM으로 예상 질문 초안을 생성하고
  - 사람이 검토하면서 테스트셋을 다듬음
- 측정 기준:
  - `top-5 기준`으로 정답 문서 포함 여부를 봄

이건 꽤 괜찮은 근거다. 완벽한 검색 평가셋은 아니어도, 적어도 `감으로 좋아졌다`가 아니라 **의도적으로 만든 평가셋으로 before/after를 비교했다**고 말할 수 있다.

### 임베딩 비교 근거

- OpenAI 계열 임베딩
- BGE 계열 임베딩
- 최종 판단:
  - `한국어 FAQ 데이터셋에서 더 잘 작동하는 모델`을 선택

즉, 임베딩 선택도 그냥 하나를 고정 사용한 게 아니라 **후보 모델 비교를 통해 선택했다**고 설명 가능하다.

### Qdrant 활용 근거

- metadata filter 사용
- 질문에서 키워드를 추출한 뒤 필터링에 활용

이 부분은 `semantic retrieval만으로 해결되지 않는 실패를 metadata filtering으로 보완했다`는 식으로 표현하면 좋다.

---

## 4-2. 이 근거를 바탕으로 더 자신 있게 말할 수 있는 것

### 보강된 문장

`검색 품질 개선 과정에서 문서 기반 테스트셋을 직접 구축하고, LLM으로 예상 질문 초안을 만든 뒤 검토해 평가셋을 정제했습니다. top-5 Hit Rate 기준으로 OpenAI, BGE 계열 임베딩 후보를 비교했고, 한국어 FAQ에서 더 잘 작동하는 모델을 선택했습니다. 이후 질문 재정의·질문 분해·키워드 기반 metadata filtering을 결합해 retrieval 성능을 개선했습니다.`

### 면접에서 말할 수 있는 포인트

- 검색 품질을 개선할 때 먼저 `평가할 기준`부터 만들려고 했다
- 도메인 특화 질문은 일반 임베딩만으로 잘 안 잡혀서, 키워드 추출 + metadata filter를 같이 썼다
- 모델 하나 바꾸는 문제로 보지 않고 `질의 구조화 + 모델 선택 + 검색 전략`의 조합으로 풀었다
- 평가는 `top-5 Hit Rate` 기준으로 봤고, 한국어 FAQ 환경에 맞는 모델을 선택했다

---

## 4-3. 그래도 아직 조심해야 하는 것

지금 근거가 생겼어도 여전히 아래 표현은 조심해야 한다.

- `검색 랭킹 모델을 직접 개발했다`
- `검색 relevance optimization을 로그 기반으로 체계적으로 수행했다`
- `검색 로그 기반 온라인 실험으로 랭킹을 고도화했다`

지금 확보된 근거는 어디까지나 **retrieval 평가셋 구축, 임베딩 비교, metadata filter를 활용한 retrieval 개선** 쪽이다.

---

## 5. 지금 기준 내 솔직한 평가

### 강점

- 검색 품질 문제를 추상적으로 말하지 않고 실패 패턴 단위로 설명할 수 있음
- retrieval 성능을 올리기 위해 query understanding 단계를 설계한 경험이 있음
- 문서 기반 테스트셋을 직접 만들고 LLM 초안 + 사람 검토로 평가셋을 구축했다는 점은 좋음
- top-5 Hit Rate 기준으로 OpenAI, BGE 계열 임베딩 후보를 비교했다는 근거가 있음
- 한국어 FAQ 데이터셋에 맞는 모델을 선택했다는 설명이 가능함
- metadata filter에 키워드 추출을 붙여 검색 실패를 보완했다는 설명이 가능함
- 검색 품질을 실제 사용자 경험과 운영 지표에 연결해서 설명 가능

### 약점

- 전통적인 검색 시스템 문법(BM25, inverted index, LTR, ranking feature engineering) 경험 근거가 약함
- 검색 로그 기반 relevance tuning을 체계적으로 했다는 증거가 부족함
- 규모 측면에서 숨고의 서비스 검색과 바로 1:1 대응된다고 말하긴 어려움

### 현재 결론

`검색 모델 개선`을 전면 메인 경험처럼 내세우기보다, `retrieval 품질 개선과 query understanding 경험`으로 정직하게 연결하는 것이 맞다.

## 6. 부족한 부분을 공부해서 채울 주제

숨고 과제나 면접 대비로 보완하려면 아래 순서가 현실적이다.

### 1단계. 검색 문제를 구조적으로 설명하는 언어 익히기

- Query Understanding
- Retrieval
- Ranking
- Reranking
- Precision / Recall / Hit Rate / NDCG / MRR
- Online metric vs Offline metric

### 2단계. 전통 검색 기본기 보완

- BM25
- inverted index
- lexical search vs semantic search
- hybrid retrieval

### 3단계. 랭킹 개선 기본기 보완

- learning to rank 개념
- pointwise / pairwise / listwise
- 랭킹 feature 예시
- 클릭 로그 편향 문제

### 4단계. 숨고 맥락으로 확장해서 생각해보기

- 고객이 어떤 검색어를 치는가
- 의도 파악이 어려운 짧은 검색어를 어떻게 다룰 것인가
- 서비스/고수/콘텐츠를 한 검색면에 같이 보여줄 때 랭킹을 어떻게 설계할 것인가
- 리뷰/채팅/전환 데이터를 검색 품질 개선에 어떻게 연결할 것인가
