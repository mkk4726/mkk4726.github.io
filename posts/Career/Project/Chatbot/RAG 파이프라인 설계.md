## 배경

기본 RAG 파이프라인(사용자 질문 → 임베딩 → 벡터 검색 → LLM 답변 생성)으로는 병원 도메인 질의에 대해 제대로 검색과 답변이 되지 않았습니다.

주요 실패 패턴은 다음과 같았습니다.
- **다턴 대화의 맥락 단절** : "라섹은?" 같은 짧은 질문은 맥락 없이는 검색이 불가능
- **복합 질의의 검색 실패** : "라식과 라섹의 가격 알려줘" → 두 정보를 모두 포함한 문서가 없으면 실패
- **도메인 특화 단어 미인식** : 원장 이름, 수술명 등을 임베딩 모델이 모름

## 1. 대화 맥락을 고려한 질문 재정의 (Query Rewriting)

### 문제

사용자는 이전 대화 내용을 참조하는 **간략한 질문**을 자주 던집니다. 이를 맥락 없이 그대로 문서 검색에 사용하면 정확한 답변을 제공할 수 없습니다.

```
Q1: 라식 가격은 얼마야?
A1: 라식 가격은 ~~
Q2: 라섹은?  ← 단독으로는 불완전한 질문
```

Q2를 검색에 그대로 쓰면 "라섹이 뭐야?"로 해석되어 엉뚱한 문서가 검색됩니다. 맥락상 "라섹의 가격은 얼마야?"로 재정의해야 합니다.

### 해결

- **Redis**로 세션별 짧은 기간의 대화기록 관리 (TTL로 자동 만료)
- 새 질문이 들어오면 해당 세션의 최근 대화기록을 LLM에 전달 → **완전한 형태의 질문으로 재정의**하도록 프롬프트 설계

Redis를 선택한 이유는 빠른 읽기/쓰기 속도와 TTL 기능 때문이었습니다.

### 효과

- 불완전한 질문이나 대명사가 포함된 질문도 맥락에 맞는 완전한 질문으로 변환
- 특히 **3턴 이상 이어지는 대화에서 검색 정확도가 크게 향상**

## 2. 복합 질문 쪼개기 + 키워드 추출 (Query Decomposition + Keyword Extraction)

### 문제

사용자는 종종 하나의 질문에 **여러 의도**를 담습니다.
- "라식과 라섹의 가격 알려줘" → "라식 가격", "라섹 가격" 각각 검색해야 함
- 하나로 검색하면 두 정보를 모두 포함한 문서가 없을 경우 실패

또한 **임베딩 모델의 한계**가 있었습니다. 사전 학습된 모델은 일반적인 언어 패턴은 잘 이해하지만, **특정 병원의 원장 이름, 수술명 같은 도메인 특화 용어는 제대로 인식하지 못했습니다.**

### 해결

**Query Decomposition**
- LLM 프롬프트로 질문을 여러 독립 질문으로 분해
- 예: "라식과 라섹의 가격 및 회복기간 알려줘" → 4개 질문으로 분해
  - "라식의 가격", "라섹의 가격", "라식의 회복기간", "라섹의 회복기간"
- 각 질문에 대해 문서 검색 후 결과 통합

**Keyword Extraction**
- 질문에서 핵심 키워드(원장명, 고유명사 등) 추출
- 문서의 메타데이터/본문에 해당 키워드를 포함한 문서를 **필터링해서 검색**
- 예: "김민규 원장님에 대한 정보 알려줘" → "김민규" 키워드 추출 → 메타데이터 필터링

이는 **시멘틱 서치 + 키워드 기반 검색을 결합한 하이브리드 검색** 구조입니다. 의미적 유사도와 정확한 키워드 매칭의 장점을 모두 활용할 수 있었고, 특히 FAQ 데이터셋이 키워드 중심으로 구성되어 있어 효과가 컸습니다.

---

## 3. 임베딩 모델 선정 : 평가 데이터셋 직접 구축

### 문제

아무리 앞단 전처리(재정의/쪼개기/키워드)를 잘 해도, 결국 문서 검색 성능을 좌우하는 핵심은 **시멘틱 서치에 사용될 임베딩 벡터의 품질**입니다.

시중엔 다양한 임베딩 모델이 있지만, 각 모델마다 성능과 특성이 다릅니다. **한국어 FAQ 데이터셋**이라는 특수 환경에 적합한 모델을 객관적으로 선정할 필요가 있었습니다.

### 해결

**1) 평가용 데이터셋 직접 구축**
- FAQ 데이터의 각 답변에 대해, LLM으로 **사용자가 실제로 물을 법한 다양한 형태의 질문들**을 자동 생성
- 이렇게 구성한 테스트셋으로 여러 임베딩 모델 비교 평가

**2) 모델 선정**
- 검색 정확도(Hit Rate@k)가 가장 높은 모델을 선택

**3) Vector DB 선정 : Qdrant**
- 고성능 벡터 검색 + 우수한 필터링 기능 → 키워드 기반 필터링과 시멘틱 서치 동시 수행에 적합
- **멀티테넌시 기능**을 활용해 FAQ 문서 버전별로 독립적인 공간 관리
- FAQ 문서 수정/추가 시 자동으로 벡터 재생성되는 파이프라인 구축 → 유지보수 비용 최소화

---

## 4. 답변 생성 (Answer Generation)

### 문제

검색된 문서를 바탕으로 최종 답변을 생성할 때, 두 가지를 동시에 달성해야 했습니다.
- **정확성** : RAG의 가장 큰 문제점인 환각(hallucination) 최소화. 고객 상담 도메인 특성상 잘못된 정보는 신뢰도 문제로 직결
- **사용자 경험** : 문서를 그대로 복사하는 게 아니라, 이해하기 쉽고 친근한 자연스러운 답변

### 해결 : 프롬프트 엔지니어링

프롬프트에서 LLM의 역할과 제약사항을 명확히 정의했습니다.

1. **참조 문서의 내용만을 바탕으로 답변** — 문서에 없는 내용은 생성 금지. 정보 불충분 시 "관련 정보를 찾을 수 없습니다"로 솔직하게 응답
2. **친근한 톤 유지** — 상담사가 친절하게 설명해주는 듯한 톤. 전문 용어엔 간단한 부연 설명
3. **정보 통합** — 여러 문서에서 검색된 정보를 일관성 있는 하나의 답변으로 재구성 (단순 나열 X)

---

## 전체 성과

- **Hit Rate 70% 향상** (40~50점 → 90~100점)
- **검색 정확도 90% 이상** 달성
- 도메인 특화 용어에 대한 검색 누락 문제 해결
- 복합 질문 분리·처리로 실제 사용자 질의에 대한 응답 품질 향상

---

## 배운 점

검색 성능 개선은 단순히 모델의 성능 문제가 아니었습니다. **사용자의 복잡한 질의를 올바르게 정의하고 구조화하는 과정**이 핵심이었습니다. 같은 GPT, 같은 임베딩 모델을 쓰더라도 앞단에서 질문을 어떻게 가공하느냐에 따라 최종 답변 품질이 극명하게 달라졌습니다.

이 파이프라인은 이후 **글로벌 확장 시 "언어별로 복제해야 하는 유지보수 부담"**의 원인이 되었고, 이를 Agentic RAG로 해결하는 과정에서 또 다른 배움이 있었습니다. (→ [[Career/Project/Chatbot/글로벌 확장 - Agentic RAG|글로벌 확장 - Agentic RAG]])
