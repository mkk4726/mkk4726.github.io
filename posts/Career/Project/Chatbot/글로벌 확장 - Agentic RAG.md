## 배경

국내 서비스에서 어느 정도 안정화된 챗봇을 **해외 채널로 확장**해야 하는 과제가 주어졌습니다. 하지만 기존 파이프라인은 **한국어에 강하게 최적화**되어 있어 그대로 확장하기 어려웠습니다.

### 한국어 특화 파이프라인을 복제하려고 했을 때의 문제

언어별로 파이프라인을 복제하는 접근을 검토해보니, 유지보수 비용이 폭증하는 구조였습니다.

1. **언어별 FAQ 문서 관리 필요**
   - 한국어 FAQ를 각 언어로 번역/로컬라이징
   - 문서 업데이트 시마다 N개 언어로 동기화
2. **언어별 임베딩 모델 선정 / 평가 필요**
   - 각 언어에 최적화된 임베딩 모델을 다시 선정하고 평가 데이터셋도 언어별로 새로 구축해야 함
3. **언어별 프롬프트 최적화 필요**
   - 질문 재정의, 쪼개기, 키워드 추출 프롬프트를 언어별로 검증
4. **언어별 검색 최적화 기법 적용 / 확인 필요**
   - 하이브리드 검색의 키워드 추출이 언어별로 다르게 동작
   - 도메인 특화 단어 사전도 언어별로 구축

즉, **N개 언어 = N개 파이프라인**을 만들고 유지해야 하는 상황이었고, AI 인력 1명으로 이를 감당하는 것은 불가능에 가까웠습니다.

## 접근 : Agentic RAG로 구조 전환

언어별 복제 대신 **LLM이 직접 검색 계획을 세우고 도구를 호출하는 Agentic 구조**로 파이프라인을 전환했습니다.

### Agentic RAG의 핵심 아이디어

기존 RAG는 **고정된 파이프라인**이었습니다.
```
[질문 재정의] → [쪼개기] → [키워드 추출] → [검색] → [답변 생성]
```
각 단계마다 별도의 프롬프트와 로직이 존재했고, 이것들이 **언어별로 최적화되어야 하는 지점**이었습니다.

Agentic RAG는 이를 **LLM 에이전트 + Tool**로 재설계합니다.
```
[LLM Agent]
   ├── Tool: search_documents(query, filters)
   ├── Tool: get_conversation_history()
   └── Tool: ...

사용자 질문 → Agent가 스스로 판단 → 필요한 Tool 호출 → 답변 생성
```

LLM이 스스로 "이 질문을 어떻게 분해할지", "어떤 키워드로 필터링할지", "검색 결과가 부족하면 재검색할지"를 결정합니다. **언어에 독립적**인 구조가 되는 이유는, 현재의 최신 LLM이 이미 다국어 이해 능력을 내재하고 있기 때문입니다.

### 이 구조의 이점

1. **언어별 로직 분리 불필요**
   - 질문 재정의, 쪼개기, 키워드 추출이 모두 **LLM의 reasoning으로 통합**
   - 언어별 프롬프트 튜닝 없이 동일 프롬프트로 여러 언어 대응
2. **FAQ 문서만 언어별로 유지하면 됨**
   - 언어별로 별도의 모델/파이프라인을 유지할 필요 없음
3. **동적인 검색 전략**
   - 처음 검색이 부족하면 Agent가 재검색
   - 복합 질문은 Agent가 스스로 여러 번 검색해서 통합

## 설계 시 고려한 트레이드오프

Agentic 구조는 장점만 있는 것은 아니었습니다. 다음 트레이드오프를 의식적으로 관리했습니다.

| 측면 | 기존 RAG | Agentic RAG |
|------|----------|-------------|
| **응답 시간** | 고정 (짧음) | Agent의 여러 Tool 호출로 늘어남 |
| **비용** | 낮음 | Tool 호출마다 LLM 호출 → 비용 증가 |
| **디버깅** | 단계별로 추적 쉬움 | Agent의 판단 과정이 블랙박스에 가까워짐 |
| **유지보수** | 언어별로 N배 | **언어와 무관하게 1개** |

글로벌 확장이라는 목표에서는 **유지보수 비용 감소의 이득이 다른 단점을 상쇄**한다고 판단했습니다. 응답 시간과 비용은 각 Tool의 효율화와 Agent의 호출 수 제한으로 일정 수준 관리 가능했습니다.

## 성과

- **언어별 파이프라인 N개 → Agentic 단일 구조 1개**로 통합
- **신규 언어 추가 공수 약 80% 감소** (수주 → 수일 수준. FAQ 문서만 추가하면 대응 가능)
- **다국어 Hit Rate**: 한국어 파이프라인 대비 **90% 이상 수준 유지** (답변 품질 유지)
- AI 인력 1명으로 글로벌 확장을 감당할 수 있는 구조 확보

> 위 수치는 일부 정성적 판단 기반의 추정치입니다. 구체적인 측정치는 별도 평가 문서 참고.

## 배운 점

**"문제를 잘게 쪼개서 고정 파이프라인으로 푸는 방식"**과 **"LLM의 reasoning에 위임해서 동적으로 푸는 방식"**은 서로 다른 상황에 적합합니다.

- 단일 도메인, 단일 언어, 예측 가능한 질의 → **고정 파이프라인** (빠르고 저렴하고 디버깅 쉬움)
- 다국어, 다양한 질의 패턴, 유지보수 인력 제한 → **Agentic 구조** (느리고 비싸지만 확장 용이)

같은 RAG라도 **"무엇을 최적화할 것인가"**에 따라 설계가 완전히 달라진다는 것을 체감했습니다. 국내 서비스에서는 "응답 속도와 비용"이 핵심이었기에 고정 파이프라인이 맞았고, 글로벌 확장에서는 "유지보수 가능성"이 핵심이었기에 Agentic이 맞았습니다.

엔지니어로서 **"제약 조건이 무엇인지 정확히 파악하고, 그에 맞는 구조를 선택"**하는 감각을 키울 수 있었던 프로젝트였습니다.
