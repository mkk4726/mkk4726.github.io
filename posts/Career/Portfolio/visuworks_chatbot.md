## 반복되는 질의를 챗봇으로 대체해 기존 상담 40% 자동화

### 문제

병원 상담사들이 반복되는 고객 질의(운영시간, 가격, 예약 문의 등)에 시간을 소모하면서 정작 필요한 상담 업무를 하지 못하는 상황이었다.

### 해결

FAQ 데이터셋을 기반으로 RAG 파이프라인을 구축해 고객 질문에 자동으로 답변하는 챗봇을 개발했다. 답변 후 만족/불만족 피드백을 수집하고, 이를 분석해 지속적으로 개선하는 방식으로 운영했다.

### 성과

- 일평균 100~200건의 반복 상담을 자동화
- 기존 상담 업무의 **40% 자동화** 달성

## 기본 RAG 파이프라인의 검색 누락 문제를 도메인 특화 파이프라인 설계로 개선, Hit Rate 70% 향상

### 문제

기본 RAG 파이프라인으로는 병원 도메인 질의에 대해 제대로 검색과 답변이 되지 않았다. 특히:
- 다턴 대화에서 맥락이 단절되어 검색 오류 발생
- 복합 질의("라식과 라섹의 가격 알려줘")에서 검색 실패
- 원장 이름 등 도메인 특화 단어를 임베딩 모델이 인식 못함

### 해결

도메인 특화 파이프라인을 설계했다:

**1. Query Rewriting**
- Redis로 대화 히스토리를 관리하고, LLM이 맥락을 반영해 질문을 독립적으로 재구성
- "라섹은?" → "라섹의 가격은 얼마야?"로 변환

**2. Query Decomposition + Keyword Extraction**
- 복합 질의를 단위 질문으로 분해 ("라식 가격", "라섹 가격" 분리)
- 도메인 키워드(원장명, 수술명 등)를 추출해 Qdrant metadata filter에 활용
- semantic retrieval만으로 놓치는 도메인 고유명사 검색 실패를 보완

**3. 임베딩 모델 선정**
- 검색 품질을 검증할 기준이 필요하다고 판단해, 제공된 문서를 바탕으로 예상 질문 초안을 생성하고 이를 검토·정제해 평가용 테스트셋을 직접 구축
- 이 과정에서 LLM은 예상 질문 초안을 빠르게 만드는 보조 도구로 활용
- top-5 Hit Rate 기준으로 OpenAI, BGE 계열 임베딩 후보를 비교 평가
- 한국어 FAQ 데이터셋에서 더 잘 작동하는 모델 선택

### 성과

- **Hit Rate 70% 향상** (40~50점 → 90~100점)
- 검색 정확도 90% 이상 달성

## LLM 생성이 불필요한 질의를 분류 모델 기반 고정답변 처리로 평균 응답시간 30% 개선

### 문제

서비스 운영 중 불만족 피드백을 분석한 결과, 답변 정확도 문제가 아니라 "답변할 수 없는 질문에 엉뚱한 대답을 하거나 재질문만 반복하는 상황"에서 불만족이 발생했다. 예: 주민번호와 전화번호를 입력하며 예약 문의를 하는 경우.

또한 운영시간, 주차 안내 같은 단순 질의에도 LLM을 호출해 4~5초의 응답시간이 소요되었다.

### 해결

RAG 파이프라인 앞단에 분류 모델을 배치했다:
- **고정답변 대상**: 운영시간, 주차, 상담원 연결 등 → LLM 없이 즉시 응답
- **RAG 대상**: 실제 검색과 생성이 필요한 질의만 파이프라인 진입
- 전체 질의의 약 50%가 분류 단계에서 처리됨

### 성과

- 응답시간 **4~5초 → 1~2초** (고정답변 대상)
- 전체 평균 응답시간 약 **30% 단축**
- 불만족도 **80% 이상 감소**

자세한 구현: [[Career/Project/Chatbot/질문 분류 및 응답시간 개선|질문 분류 및 응답시간 개선]]

## 일평균 200건 대화기록 수동 검토를 LLM 기반 자동화로 전환, 검토 시간 83% 단축 (30분 → 5분)

### 문제

챗봇은 에러를 발생시키지 않고 조용히 실패(silent failure)하는 특성이 있어 대화 기록을 모니터링해야 했다. 하지만 일평균 150~200건의 대화 기록을 사람이 직접 검토하는 데 약 30분이 소요되었다.

### 해결

LLM 기반 자동 평가 시스템을 구축했다:

**1. 평가지표 설계**
- RAGAS 프레임워크 참고해 Context Sufficiency, Faithfulness 등 지표 정의
- 답변이 검색된 문서에 얼마나 근거하는지 점수화

**2. 자동화 파이프라인**
- Airflow로 스프레드시트 로그에서 배치 평가 실행
- 일별/주별 점수와 점수가 낮은 대화를 Slack으로 알림
- 문제 의심 사례만 선별적으로 검토

### 성과

- 모니터링 시간 **30분 → 5분** (83% 단축)
- Silent failure 조기 감지 체계 구축

자세한 구현: [[Career/Project/Chatbot/모니터링 자동화|모니터링 자동화]]

## 한국어 특화 파이프라인의 다국어 성능 한계를 Agentic RAG로 검색 구조 전환해 해결, 유지보수 비용 대폭 감소

### 문제

국내에서 안정화된 챗봇을 해외 채널로 확장하면서, 한국어 특화 파이프라인의 구조적 한계가 드러났다. 언어별로 파이프라인을 복제하려면:
- **언어별 FAQ 문서 관리** (번역, 로컬라이징, 동기화)
- **언어별 임베딩 모델 선정 및 평가 데이터셋 구축**
- **언어별 프롬프트 최적화** (재정의, 쪼개기, 키워드 추출)
- **언어별 검색 최적화 기법 적용 및 검증**

즉, N개 언어 = N개 파이프라인을 유지해야 하는 상황이었고, AI 인력 1명으로 감당 불가능한 구조였다.

### 해결

**Agentic RAG**로 구조 전환했다:
- 고정된 파이프라인(재정의 → 쪼개기 → 키워드 → 검색) 대신, **LLM 에이전트 + Tool** 구조로 재설계
- LLM이 직접 검색 계획을 수립하고 Tool(검색/필터링)을 호출
- 언어별 로직을 LLM의 reasoning에 통합 → **언어에 독립적인 파이프라인**
- FAQ 문서만 언어별로 유지하면 되는 구조 확보

### 성과

- 언어별 파이프라인 N개 → **Agentic 단일 구조 1개**로 통합
- 신규 언어 추가 공수 **약 80% 감소** (수주 → 수일, FAQ 문서만 추가)
- 다국어 Hit Rate 한국어 파이프라인 대비 **90% 이상 수준 유지**
- AI 인력 1명으로 글로벌 확장을 감당할 수 있는 구조 확보
