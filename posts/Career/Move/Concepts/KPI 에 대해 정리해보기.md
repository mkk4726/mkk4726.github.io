DS, AIE, MLE에 대한  기본적으로 생각하는 4가지 KPI

```
1순위: 비즈니스 임팩트 (매출, 전환율)
2순위: A/B 테스트 결과 (비즈니스 임팩트 증명하기 위한 장치)
3순위: 모델 성능 (offline metric)
4순위: 시스템 안정성 (MLOps)
```

# 1. 비즈니스 임팩트 지표

> 모델이 실제 사업에 기여한 결과를 측정합니다.
- **수익/비용**: 모델 도입으로 인한 매출 증가액, 비용 절감액
- **전환율**: A/B 테스트 기준 개선율 (CTR, CVR 등)
- **자동화율**: 사람이 하던 업무 중 자동화된 비율
- **추천/개인화 성과**: 추천 클릭률, 세션 시간, 구매 전환

>ML/AI 직군에서 가장 중요하면서도 가장 측정하기 어려운 축입니다. 
>핵심 질문은 **"모델이 실제로 돈/가치를 만들었냐?"** 입니다.

나는 이게 직업의 당위성(?)과도 이어진다고 생각한다. 그래서 내가 가진 직업이 왜 필요한지, 어떤 역할을 하고 있는지를 증명할 수 있어야 하고, 이걸 증명해야 높은 연봉을 받을 수 있다고 생각한다.

> [!note]- 클로드: 전체를 꿰는 핵심 개념: **Attribution (기여도 분석)**
> 비즈니스 임팩트 측정에서 가장 어려운 문제는 **"이 성과가 정말 내 모델 덕분인가?"** 입니다.
>```
>유저가 구매함
 > ← 추천 모델이 상품을 노출했고
 > ← 가격 모델이 할인을 제안했고
 > ← 이탈방지 모델이 쿠폰을 뿌렸음
> → 누구의 공?
> ```
> 
> 이를 해결하는 방법:
> 
> - **A/B 테스트**: 가장 신뢰도 높음, 하지만 항상 가능하진 않음
> - **Causal Inference (인과추론)**: 실험이 불가할 때 관측 데이터로 인과 추정
> - **Multi-touch Attribution**: 여러 모델/채널의 기여를 가중 배분
>

## 1.1 수익증가 / 비용절감
> 모델의 존재 자체가 P&L에 미치는 영향입니다.

**측정 방법**
- **A/B 테스트**: 모델 적용 그룹 vs 미적용 그룹의 매출 차이
- **Before/After**: 모델 도입 전후 동일 기간 비교 (계절성 보정 필요)
- **holdout group**: 일부 유저에게 의도적으로 모델을 끄고 기준선 유지

**예시**
> 가격 최적화 모델 도입 → 동일 판매량에서 마진 +2.3% 이상탐지 모델 도입 → 사기 거래 손실 월 5천만원 감소

**함정**
- 상관관계 ≠ 인과관계. 모델 도입 시기에 마케팅도 강화됐다면? → 반드시 **통제 실험** 필요
- 장기 효과 vs 단기 효과가 다를 수 있음 (추천 모델이 단기엔 클릭 올리지만 장기엔 피로도 유발


## 1.2 전환율 (CVR / CTR)
> 추천, 광고, 검색 등 **유저 행동을 유도하는 모델**에서 핵심 지표입니다.

특정 도메인에서 목표로하는 KPI

| 지표        | 의미            | 적용 모델     |
| --------- | ------------- | --------- |
| CTR (클릭률) | 노출 대비 클릭 수    | 추천, 광고    |
| CVR (전환율) | 클릭 대비 구매/가입 수 | 추천, 검색    |
| ARPU      | 유저 1인당 평균 수익  | 개인화 모델 전반 |

**주의점**: CTR만 높이면 낚시성 콘텐츠가 이길 수 있음. 반드시 **CVR, 만족도, 재방문율**과 함께 봐야 함.

## 1.3 자동화율
> 사람이 하던 반복 작업을 모델이 대체한 비율입니다.

**계산 방식**

```
자동화율 = 모델이 처리한 건수 / 전체 처리 건수
```

**예시**
> 고객 문의 1만 건 중 7천 건을 챗봇이 해결 → 자동화율 70% 
> 문서 분류 작업: 기존 5명이 하루 1,000건 처리 → 모델 도입 후 1명이 검수만 담당

**연계 지표**

- **인건비 절감액**: 자동화된 업무량 × 시간당 비용
- **처리 속도**: 사람 평균 처리 시간 vs 모델 처리 시간
- **에스컬레이션율**: 모델이 못 풀어서 사람에게 넘긴 비율 (낮을수록 좋음)

## 1.4 추천/개인화 성과
 > 추천 시스템에 특화된 지표 묶음입니다.

**온라인 지표** (실서비스에서 측정)

- **세션 시간**: 추천 콘텐츠 소비로 체류 시간이 늘었는가
- **클릭 다양성**: 유저가 새로운 아이템을 발견하는가 (필터버블 방지)
- **재구매율 / 재방문율**: 추천 경험이 충성도로 이어지는가

**오프라인 지표** (모델 평가 단계)

- **Precision@K**: 상위 K개 추천 중 실제 관심 아이템 비율
- **NDCG**: 순위까지 고려한 추천 품질
- **Coverage**: 전체 아이템 중 추천에 노출되는 비율

> 오프라인 지표가 좋아도 온라인 지표가 나쁠 수 있음. 둘의 **상관관계를 지속적으로 검증**하는 게 중요.

# 2.  모델 성능

이 파트에서 내가 궁금한건 평가지표를 어떻게 제시해야하지?

> [!note]- 클로드 : 평가지표 제시하는 법
> **핵심 질문: "MAE가 100이면 좋은 건가요?"**
>
> 이게 굉장히 중요한 포인트입니다. **MAE 100은 맥락 없이는 아무 의미가 없습니다.**
>
> **방법 1: Baseline과 비교**
>
> 가장 기본적이고 강력한 방법입니다.
>
> ```
> Baseline: 아무 모델 없이 "항상 평균값으로 예측"했을 때 MAE
> → Baseline MAE = 350
> → 내 모델 MAE = 100
> → 기준 대비 71% 오차 감소 ✅
> ```
>
> **자주 쓰는 Baseline 종류**
>
> - **평균 예측**: 항상 train set 평균을 예측
> - **직전값 예측**: 어제 값을 오늘 예측값으로 씀 (시계열)
> - **단순 규칙**: 도메인 전문가의 경험칙
>
> "MAE 100입니다" 보다 **"기존 규칙 기반 대비 오차 71% 감소"** 가 훨씬 설득력 있음
>
> 
>
> **방법 2: 예측 대상의 스케일과 비교**
>
> MAE는 **예측값의 단위와 같기 때문에** 스케일을 같이 제시해야 합니다.
>
> ```
> 아파트 가격 예측
>   → 평균 가격: 50,000만원 / MAE: 100만원 → 오차율 0.2% 🟢 아주 좋음
>
> 편의점 일매출 예측
>   → 평균 매출: 80만원 / MAE: 100만원 → 오차가 평균보다 큼 🔴 나쁨
> ```
>
> 이걸 수치로 표현한 게 **MAPE**이지만, MAPE도 한계가 있습니다 (정답이 0에 가까우면 폭발). 그래서 실무에선 아래처럼 직접 서술합니다.
>
> **"평균 예측 오차는 100만원으로, 실제 평균값(5천만원)의 약 2% 수준입니다."**
>
> 
>
> **방법 3: 비즈니스 허용 오차와 비교**
>
> 기술 지표를 **의사결정 기준**으로 번역하는 방법입니다.
>
> ```
> 물류 수요 예측 모델
>   → 재고 담당자: "±15% 안에만 들어오면 우리가 커버 가능해"
>   → 모델 MAPE: 8% → 허용 기준 내 ✅
>
> 전력 수요 예측
>   → "예측 오차가 500MW 초과하면 정전 리스크"
>   → 모델 MAE: 100MW → 안전 마진 충분 ✅
> ```
>
> 이 방법이 보고서에서 가장 임팩트 있습니다. **숫자가 아니라 결과로 말하는 것.**
>
> 
>
> **방법 4: 이전 모델 / 경쟁 모델과 비교**
>
> ```
> v1 모델 MAE: 180
> v2 모델 MAE: 100  → 44% 개선
> 업계 논문 SOTA MAE: 85  → SOTA 대비 18% 차이
> ```
>
> 
>
> **정리: MAE 100을 보고서에 제시하는 법**
>
> |나쁜 제시|좋은 제시|
> |---|---|
> |"MAE = 100입니다"|"평균값(5,000) 대비 오차 2% 수준"|
> |"성능이 좋습니다"|"Baseline 대비 오차 71% 감소"|
> |숫자만 나열|"담당자 허용 기준(±15%) 내인 8% 달성"|
> |단독 수치|"전 버전 MAE 180 → 100으로 44% 개선"|
>
> **한 문장 원칙**: 숫자는 반드시 **기준점(Baseline, 스케일, 허용치, 이전 버전)** 과 함께 제시한다.
>
> 
>
> **방법 5: 통계적 신뢰구간과 함께 제시**
>
> 단일 숫자만 던지면 "운 좋게 나온 거 아닌가?" 라는 반론에 취약합니다.
>
> ```
> 모델 MAE: 100 ± 12 (95% CI, 5-fold CV 기준)
> → 최악의 경우에도 MAE 112 이하
> ```
>
> - **K-fold Cross Validation** 결과의 평균 ± 표준편차
> - **Bootstrap** 으로 신뢰구간 산출
> - A/B 테스트 결과라면 **p-value** 또는 **신뢰구간**을 반드시 포함
>
> 이렇게 하면 "재현 가능한 성능"임을 증명할 수 있습니다.
>
> 
>
> **분류 모델에도 같은 원칙 적용**
>
> 위 방법 1~5는 MAE 예시지만, 분류 지표(Precision, Recall, F1 등)에도 동일하게 적용됩니다.
>
> |나쁜 제시|좋은 제시|
> |---|---|
> |"F1 = 0.85"|"기존 규칙 기반(F1 0.62) 대비 37% 개선"|
> |"Recall 95%"|"실제 사기 건수 대비 95% 탐지, 미탐지 건당 손실 200만원 → 월 손실 1천만원 이내로 억제"|
> |"AUC 0.92"|"랜덤(0.5) 대비 0.42p 향상, 업계 SOTA(0.94) 대비 0.02p 차이"|
>
> 특히 분류 문제에서는 **어떤 지표를 왜 선택했는지**를 먼저 설명해야 합니다.
>
> ```
> 암 진단 → Recall 우선 (놓치면 생명 위험)
>   → "Recall 95% 달성, 위양성(FP)은 추가 검사로 커버 가능"
>
> 스팸 필터 → Precision 우선 (정상 메일 차단이 더 치명적)
>   → "Precision 98%, 전체 스팸 중 12%만 통과 (사용자 체감 미미)"
> ```
>
> 
>
> **실전 보고 템플릿**
>
> ```
> [지표 선택 근거] 본 문제는 ___이므로 ___를 핵심 지표로 선정
> [절대 성능]      모델 ___: XX (95% CI: XX~XX)
> [상대 비교]      Baseline 대비 YY% 개선 / 이전 버전 대비 ZZ% 개선
> [비즈니스 해석]   이는 _____ 기준으로 허용 범위 내이며,
>                 월 약 ___원의 비용 절감 / 매출 증가에 해당
> ```

- 비교해서 제시하기
baseline 대비 몇%향상 이런식으로.

- 비즈니스 오차 제시
렌즈 사이징으로 예시를 들면, vaulting 예측의 경우 MAE 100을 이야기하신게 허용할 수 있는 수준이라서.
그러니까 표시할 때 MAE100보다는 비즈니스 허용 기준 90% 달성. 이런식으로 풀어서 쓸 수 있겠네

- MAPE 사용하기
스케일로 나눠놓은 것이니까, 이거 제시하면 더 명확히 보이겠네

- 통계적 신뢰구간을 같이 제시하면 더 신뢰도 있는 평가지표가 될 듯
신뢰도를 주기 위해서를 떠나서, 내가 모델을 평가할 때도 이런 관점에서 평가를 할 필요가 있겠네.

# 3. 엔지니어링, 운영 지표

>[!note]- 클로드 설명
> 모델 성능이 아무리 좋아도 **프로덕션에서 안정적으로 돌아가지 않으면 의미가 없다.** 
> 이 축은 **「모델을 서비스에서 잘 운영하고 있는가?」** 를 측정한다.
>
> **3.1 레이턴시 (Latency)**
>
>추론 요청 한 건을 처리하는 데 걸리는 시간이다.
>
>**왜 평균이 아니라 p95·p99인가**
>
>느린 소수 요청이 평균을 가려서, 체감 품질과 맞지 않을 수 있다.
>
>```
>요청 100건의 응답 시간:
>  98건 → 50ms
>  2건  → 3,000ms (타임아웃 직전)
>
>평균: 약 110ms → 겉으로는 양호해 보임
>p99: 3,000ms   → 사용자 1%는 3초 대기
>```
>
>실무에서는 SLA를 **p95, p99** 기준으로 잡는 경우가 많다.
>
>**SLA 예시**
>
>```
>추천 API: p99 < 200ms
>실시간 사기 탐지: p99 < 50ms   (결제 흐름에 포함되면 더 엄격)
>배치 예측: N시간 내 배치 완료
>```
>
>**레이턴시를 줄이는 방향** (참고): 모델 경량화(Quantization, Pruning), 동일 입력 캐싱, CPU → GPU·전용 가속기 등.
>
> **3.2 처리량 (Throughput)**
>
>단위 시간당 처리 가능한 요청 수다. 보통 **RPS(Requests Per Second)** 로 표현한다.
>
>```
>Throughput = 초당 처리 요청 수 (RPS)
>```
>
>**레이턴시와의 관계**
>
>- 배치로 묶어 처리하면 **처리량은 늘고**, 개별 요청 기준 **지연은 커질 수 있다.**
>- 요청마다 즉시 처리하면 **지연은 줄고**, 같은 자원으로 **처리량은 상대적으로 작을 수 있다.**
>
>트래픽 피크에 **처리량 한계**에 닿으면 레이턴시가 급증한다. **부하 테스트(Load Test)** 로 한계 RPS를 미리 아는 것이 중요하다.
>
> **3.3 모델 드리프트 (Model Drift)**
>
>배포 이후 시간이 지나며 **모델 성능이 서서히 나빠지는 현상**이다. 눈에 잘 띄지 않아 위험하다.
>
>**종류**
>
>```
>Data Drift (데이터 드리프트)
>  → 입력 분포가 학습 시기와 달라짐 (환경·행동 변화)
>
>Concept Drift (컨셉 드리프트)
>  → 입력과 정답(또는 최적 행동)의 관계 자체가 바뀜
>```
>
>**감지에 쓰는 방법**
>
>| 방법 | 설명 |
>|------|------|
>| PSI | 입력 피처 분포 변화를 수치화 |
>| KL Divergence | 두 분포 간 차이 |
>| 예측값 분포 모니터링 | 출력 분포가 한쪽으로 쏠리면 이상 신호 |
>| Ground truth와 주기적 비교 | 라벨이 있으면 MAE·Accuracy 등 추이 추적 |
>
> **3.4 재학습 파이프라인 안정성**
>
>드리프트 대응이나 정기 갱신을 위해 **재학습 → 검증 → 배포** 파이프라인이 돌아가야 한다.
>
>**측정 예**
>
>```
>파이프라인 성공률 = 성공 실행 수 / 전체 실행 수
>파이프라인 리드타임 = 데이터 수집 ~ 배포까지 소요 시간
>재학습 주기 준수율 = 예정대로 실행된 비율
>```
>
>**자동화 성숙도 (참고)**
>
>```
>Lv1 수동     → 이슈 인지 후 사람이 재학습
>Lv2 스케줄   → 주기적 자동 재학습
>Lv3 트리거   → 드리프트 등 조건으로 자동 재학습
>Lv4 자동배포 → 검증 통과 시 자동 배포까지 (MLOps 고성숙)
>```
>
> **3.5 인프라 비용 (Cost Efficiency)**
>
>비즈니스 임팩트 대비 **자원을 얼마나 효율적으로 쓰는지**다.
>
>```
>비용 효율 ≈ 비즈니스 임팩트(매출·절감 등) / 인프라 비용
>```
>
>**자주 보는 지표**
>
>- **GPU 활용률**: 지속적으로 낮으면 스케줄링·인스턴스 크기 조정 검토.
>- **요청당 추론 비용**: 트래픽 증가 시 선형인지, 비용이 폭증하는지.
>- **배치 vs 실시간**: 실시간 추론이 배치보다 비싼 경우가 많아, **꼭 실시간이 필요한지** 재검토.
>
> **3.6 실무에서의 한 화면 예시**
>
>운영 지표는 보통 **대시보드**에서 함께 본다.
>
>```
>┌─────────────────────────────────────────┐
>│  모델: 구매 예측 v3.2                     │
>├──────────────┬──────────────────────────┤
>│ 레이턴시 p99  │ 143ms ✅ (기준: <200ms)  │
>│ Throughput   │ 1,200 RPS ✅             │
>│ Data Drift   │ PSI 0.18 ⚠️ (기준: <0.2) │
>│ 파이프라인    │ 마지막 성공: 2시간 전 ✅   │
>│ 인프라 비용   │ $1,240/일 (전주 대비 +3%)│
>│ 모델 MAE     │ 102 ✅ (배포 시: 98)      │
>└──────────────┴──────────────────────────┘
>```
>
>**지표 연결 (요약)**
>
>```
>드리프트 발생
>    ↓
>오프라인·온라인 성능 악화 (성능 지표로 포착)
>    ↓
>재학습 파이프라인 트리거
>    ↓
>새 모델 배포 → 레이턴시·비용 재확인
>    ↓
>비즈니스 임팩트 회복
>```


엔지니어링 부분에서 성과를 냈다고 말하려면 이정도는  적어야 한다.
