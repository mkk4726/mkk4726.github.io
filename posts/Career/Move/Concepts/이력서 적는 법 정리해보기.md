# 1. 이력서 기본기 (생각 등대)

> 이력서를 보는 사람을 고려하자

내가 제출한 이력서를 결국 같은 현업자가 검토하게 됨.
이런 사람들이 이력서를 어떤 마음으로 볼지 생각해보는게 굉장히 중요하다.
( 생각등대, 인프런 멘토, 이형 등등 공통적으로 이야기하는 부분 )

면접관은 시간이 없고 에너지도 없다 → 이력서는 빠르게 판단될 수 있도록 읽기 좋은 문서가 되어야 한다.

![[Pasted image 20260415225938.png|571]]

1. 제목 → “이름 이력서”
2. 사진 → 회사 분위기 맞게. 같이 일하고 싶은 느낌. 좋은 인상을 주도록
3. 지원동기 → 해당팀에 어떤 도움을 줄 수 있는지 적기. 채용공고 참고
4. 자기소개 → 팩트 기반으로 쓰기.
5. 프로필 → 나에 대한 정보가 한눈에 들어오도록.

![[Pasted image 20260415225958.png|576]]

1. 프로젝트 개수 → 3~4개
2. 제목, 일정, 기술 스택, 참여 인력, 서비스 카테고리 → 프로젝트가 한눈에 이해되도록
    1. 제목 → 제목만 봐도 어떤 프로젝트인지 이해가 되도록
    2. 일정 → 월단위 , 3개월 내외
    3. 기술 스택 → 프로젝트의 핵심 기술을 이해할 수 있도록
    4. 참여인력 → 구성인원 수
    5. 서비스 개요 → 어떤 도메인에서 어떤 문제를 어떻게 해결했고, 그 겨로가가 어떻게 나왔는지
3. 내가 기여한 부분
![[Pasted image 20260415230042.png|592]]
- “문제 + 해결 + 결과 + 도메인”

> 면접관이 이 한줄을 읽었을 때 이런 생각이 무의식적으로 떠올라야 합니다. ”어떤 도메인(기능)에서 어떤 문제를 어떻게 해결했고, 그 결과가 어떻게 나왔는가” 이 느낌이 바로 박혀야 한다는 거죠. 도메인이라고 함은 ‘어떠한 기능’이라고 생각하시면 이해하기 쉽습니다.

> 즉, 읽는 순간 이해되고, 해결 방식과 결과가 설득력 있게 느껴지도록 작성해야 합니다. 이렇게 하나의 프로젝트에 문제 해결이 3개 정도 있는 것이 가장 좋은 방향성이고 그걸 최상단에 써주시면 됩니다.


4. 어느 정도 수준의 역량까지 적어야 할지
- 주니어 → 기본기 60% + 본인의 강점 40%    
> 면접관 입장에서 보면, 생각보다 많은 신입 분들이 세련되고 화려한 기술이나, 어렵고 복잡한 문제 해결에 대해서는 잘 이야기하는 편입니다. 그런데 기본기에 대해 질문하면, 정말 기본적인 내용인데도 불구하고 모르시는 분들이 꽤 많습니다. 면접관 입장에서는 이런 경험이 이미 많이 쌓여 있을 거에요.

# 2. 포트폴리오 기본기 (생각 등대)

1. 제목 → 이력서의 문제 해결 역량

![[Pasted image 20260415230102.png|622]]
- 이력서에 적었던 문제 해결 역량 문장들 중 핵심이 되는 부분들을 뽑아서 작성

![[Pasted image 20260415230116.png|626]]    
- 여기 보이는 제목 → 이력서 문제 해결 역량에 적혀있는 제목

2. 그림 → 전체저인 아키텍처.
    1. 모든 문제 해결 항목에 반드시 그림을 포함시키기
    2. 아이콘 사용하기 보다는 프레임워크 이름 적어주기. 한눈에 보이도록
    3. 캡처하기 보다는 따로 그리기

3. 문제 해결 → 풀어서 작성하기

4. 보편적으로는 9줄 내외면 충분

> LLM에게 요청하기 ”이 데이터를 바탕으로 문제, 해결, 결과를 각각 3줄씩 작성해서 문제 해결 흐름을 정리해줘. 중간에 애플리케이션 레벨의 내용까지 포함해도 좋아.    
- 테스트 결과를 결과에 적어주면 더 좋다.

> 모든 프로젝트를 문제 해결 관점으로 풀어낼 수 있다.
- 프로젝트를 기획하고 해결했던 과정이 모두 문제 해결의 과정
- 진행하는 과정에서 없었더라도 개선할 수 있는 포인트들을 잡아서 실행해보기
