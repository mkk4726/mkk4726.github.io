AI 서비스를 통해 분석 결과가 의사에게 전달되고, 검안사는 이를 바탕으로 주문을 실행할 수 있는 시스템을 구축해 의사결정 단계를 축소 했습니다. 이러한 프로세스 최적화를 통해 현장의 생산성을 향상시킬 수 있었습니다.

< 4.1 문제 정의 - 파편화된 데이터에서 발생하는 비효율성, AI 서비스를 도입의 어려움 > 시력교정술 수술을 위해서는 다양한 장비에서 다양한 검사를 진행합니다. 이를 통해 4~5종류의 장비별로 4~8장정도의 검사결과 이미지가 생성됩니다. 이러한 형태의 데이터는 실무자가 결과를 한눈에 보기 어려웠고, 참고하는 수식들에 입력하기 위해 해당 결과들을 확인해 직접 입력해야 했습니다. 이는 상담 혹은 수술에 이르는 과정에 비효율성을 야기했습니다. 이미지 형태로 파편화된 데이터는 업무를 방해하는 pain point였습니다. 또한 AI 서비스를 현장에 도입하려고 할 때의 큰 걸림돌 중 하나였습니다. AI 서비스를 사용하기 위해 이미지에서 값들을 찾아 입력하는 과정은 실무자들에게 큰 불편함으로 다가오고, 이는 서비스의 실패로 이어질 수 있습니다.

< 4.2 기술적 구현 - 현장의 워크플로우를 방해하지 않는 설계와 정확도 99%이상의 안정적인 OCR 파이프라인 개발 > 파이프라인을 개발하면서 가장 중요하게 생각한 부분은 실무자의 워크플로우를 방해하지 않으면서 기술을 적용하는 것입니다. 이미지를 서버에 업로드하는 것처럼 별도의 과정을 추가하게 된다면 현장의 번거로움을 덜어내고자 하는 목적을 해친다고 판단했기 때문입니다. 따라서 실무자와의 인터뷰를 통해 워크플로우를 이해하고 분석해 설계했습니다. 검사가 완료되면 검사 장비의 이미지가 관리 프로그램으로 전송되는 중간 경로에 파이프라인을 위치시켜, 데이터가 백그라운드에서 자동으로 서버에 전송되도록 했습니다. 이를 통해 실무자는 평소처럼 검사를 마치는 것만으로도 이미지 데이터가 파이프라인을 통해 처리될 수 있게 되었 정확도를 확보하는 과정에서는 검사 결과 이미지가 정형화된 포멧을 가진다는 특징을 전략적으로 활용했습니다. 텍스트의 위치를 찾는 일반적인 OCR과 달리, 고정된 레이아웃임을 활용해 Text Detection 단계를 규칙 기반으로 구현해 속도와 정확도를 모두 향상시켰습니다. 그리고 Text Recognition에 특화된 모델을 도입하고, 이미지 전처리와 결과 후처리 로직을 통해 목표하는 성능을 달성할 수 있었습니다. 결과적으로 실무자의 워크플로우에 별다른 과정을 추가하지 않고 데이터를 통합할 수 있는 OCR 파이프라인을 개발할 수 있었습니다.

< 4.3 성과 - 현장의 워크플로우 최적화와 AI 서비스 적용 환경 설정 > 이러한 파이프라인 구축 결과로 파편화되어 있던 검사 이미지 데이터를 통합된 형태로 관리할 수 있게 되었고, 이에 기반한 데이터 파이프라인을 통해 현장의 워크플로우를 최적화할 수 있었습니다. 또한 이는 현장에 AI 서비스를 적용할 수 있는 환경이 되었습니다.

[5. 마무리 - 기술의 화려함보다는 실제 현장에서 가치를 만드는 데이터 과학자] 제가 중요하게 생각하는 건 컴퓨터 앞에 앉아 모델의 지표를 높이는 일이 아닙니다. 실제 현장과 소통하며 문제를 이해하고 이를 제가 가진 기술로 해결하며 가치를 만드는 일입니다. 3번의 End-to-End로 프로젝트를 진행하면서 기술적인 깊이뿐만 아니라 실전 감각을 크게 키울 수 있었습니다. 현장의 실무자들과 인터뷰를 하며 pain point를 찾아내고 이를 기술적인 언어로 재정의하며 AI 서비스를 설계하는 과정을 경험하며, 소통하는 법, 현장의 언어와 기술적인 언어 사이의 회색지대를 채우는 법, AI 서비스가 진짜 가치를 만들기 위해 고려해야하는 것들을 배울 수 있었습니다. 대한민국 뷰티와 라이프스타일의 최전선에 있는 올리브영은 방대한 온·오프라인 데이터를 보유하고 있으며, 그만큼 해결해야 할 흥미롭고 복합적인 pain point가 존재할 것이라고 생각합니다. 저는 단순히 주어진 데이터를 분석하고 모델을 만드는 역할만을 해오지 않았습니다. 매장 매니저부터 플랫폼 기획자까지, 현장의 이해관계자들과 적극적으로 소통하며 pain point를 발굴하고, 이를 해결할 수 있는 AI 서비스를 기획하고 개발해 실질적인 가치를 만들 수 있다고 확신합니다.

[1. 자기소개 – 문제를 해결하기 위한 시스템을 설계하는 데이터 과학자] 저는 비즈니스 현장에서 발생하는 모호한 문제를 구조화하고, 이를 AI 서비스로 구현해 온 데이터 과학자 김민규입니다. 단순히 모델을 개발하는 역할에 머무르지 않고, 문제 정의부터 데이터 설계, 모델링, 서빙, 모니터링까지 End-to-End로 책임지는 역할을 수행해왔습니다. ML/DL 기반 예측 모델 개발, LLM/RAG 아키텍처 설계 및 운영 경험을 보유하고 있으며 각 문제의 특성에 맞는 기술을 선택해 시스템을 설계해왔습니다. 저는 모든 문제를 복잡하고 화려한 모델로 해결하려 하지 않습니다. 현장의 실무자와의 인터뷰를 통해 문제를 이해하고 문제를 해결하기 위한 최선의 선택을 고민해왔습니다. 고도화된 DL 모델보다 Rule 기반 접근이 적절하다면 이를 선택했습니다. 이처럼 과도한 엔지니어링을 지양하고, 문제에 가장 적합한 구조를 설계해 문제를 해결해왔습니다. Forward Deployed Data Scientist는 현장에서 문제를 정의하고, 적절한 AI 기술을 선택해 설계부터 운영까지 책임지는 역할이라고 이해하고 있습니다. 저는 LLM, RAG, ML/DL을 개별 기술이 아닌 문제 해결 도구로 활용하며 End-to-End로 구현해온 경험을 바탕으로 해당 역할을 수행할 역량을 갖추고 있습니다.

[2. 고객상담용 챗봇 서비스 개발 - RAG 아키텍처 설계 및 운영 고도화 경험] 상담 업무 자동화를 통한 상담업무 효율 향상을 위해 RAG 기반 챗봇 시스템을 설계하고 운영했습니다. 단순히 VectorDB와 LLM을 연결하는 수준이 아니라, 현장 요구사항을 반영한 아키텍처 설계와 검색 성능 고도화, 서비스 운영 그리고 LLM 기반 모니터링 자동화 구축까지 전 과정을 주도했습니다. 초기에는 문서 검색 + LLM Generation 구조의 기본 RAG 파이프라인을 구현했지만, 실무자와의 인터뷰를 통해 기술적인 요구사항을 정의하면서 모든 질의가 검색-생성을 필요로 하지 않는다는 점을 알 수 있었습니다. 이를 만족하기 위해 분류 모델을 추가한 아키텍처를 설계하고, 검색성능을 고도화해 성공적인 서비스를 구현할 수 있었습니다. 또한 대화기록 모니터링을 자동화 할 수 있는 LLM 기반 평가 파이프라인을 설계하여 운영 효율성과 품질 안정성을 동시에 확보했습니다. 단순한 RAG 기능구현이 아닌, 서비스를 온전히 운영할 수 있는 시스템을 구축했고 이를 통해 챗봇 서비스를 온전히 운영할 수 있었습니다.


< 2.1 문제에 맞는 RAG 아키텍처 설계 > 기본적인 RAG 파이프라인은 질의가 들어오면 관련 문서를 검색하고, 이를 LLM에 함께 전달해 답변을 생성하는 단일 흐름으로 구성됩니다. 그러나 현장 인터뷰를 진행하면서, 모든 질의가 검색-생성 단계을 필요로 하지 않는다는 점을 알 수 있었습니다. 예를 들어 주차장 안내나 운영시간과 같은 문의는 고정된 답변이 가장 정확하고 효율적이었으며, “상담사 연결”과 같이 명확한 액션 의도가 포함된 질의는 생성형 답변이 아니라 버튼 제공이나 연결 프로세스로 즉시 전환하는 것이 더 적절했습니다. 그럼에도 기존 구조에서는 모든 질의가 동일하게 검색-생성을 거치면서 불필요한 LLM 호출이 발생했고, 이는 응답 지연과 비용 증가, 그리고 생성 모델 특유의 부정확한 응답 가능성으로 이어질 수 있는 구조적 한계가 있었습니다. 이 문제를 해결하기 위해 파이프라인의 맨 앞 부분에 질의 분류 단계를 추가해 처리 경로를 분기하는 구조로 아키텍처를 재설계했습니다. few-shot 기반 LLM을 활용해 질의를 의도(Intent) 기준으로 분류하고, 검색·생성이 필요한 경우에만 RAG를 수행하도록 설계했습니다. 반면 고정 응답이 적절한 질의는 사전에 정의된 응답을 반환하고, 상담사 연결 의도가 명확한 경우에는 즉시 해당 프로세스로 전환하도록 흐름을 분리했습니다. 이를 통해 불필요한 검색-생성을 줄여 비용과 응답속도를 개선했고, 생성형 모델이 답변하지 않아야 할 상황을 구조적으로 차단해 운영 안정성을 높일 수 있었습니다. RAG를 단순히 적용하는 것이 아니라, 문제와 요구사항을 이해하고 이에 맞게 파이프라인을 구조적으로 설계하는 것이 LLM 시스템의 성공 여부를 결정하는 핵심요소라고 생각합니다.

< 2.2 검색 성능 고도화 > 초기 검색 구조는 embedding 기반 semantic search로 설계했습니다. 그러나 단순한 유사도 비교만으로는 목표 정확도를 확보하기 어려웠습니다. 다음과 같은 문제들이 있었습니다. 첫번째로 의료진 이름, 시술명, 의학 용어 등 고유 단어들이 임베딩 공간에서 충분히 반영되지 않아 검색이 되지 않는 문제가 있었습니다. 모델을 fine-tuning 하는 대신 비용 대비 효율을 고려해 키워드 추출 기반 hybrid 구조를 설계했습니다. 질의에서 핵심 단어를 추출해 필터링 및 가중치로 적용함으로써 정확도를 개선할 수 있었습니다. 두번째로 “라식과 라섹의 가격과 수술방법 차이” 등과 같은 복합 질의에 대해서 검색이 안되는 문제가 있었습니다. 이를 해결하기 위해 Query Decomposition을 도입해 질의를 작은 질의로 분해하고, 질의별로 검색 후 결과를 통합하는 방식으로 재구성했습니다. 셋번째로 대화의 맥락을 고려해 문서를 검색하지 못하는 문제가 있었습니다. “라식이 가격 알려줘”라고 물은 후 “라섹은?”이라는 질문에는 “라섹의 가격”에 대한 문서를 해야하지만, 단순히 질의로 문서를 검색하는 방식에서는 한계가 있었습니다. 이를 해결하기 위해 Query Rewriting 단계를 추가해 대화기록을 반영해 질의를 재작성하고 문서를 검색하도록 했습니다. 이 과정을 통해 검색 파이프라인을 “질의 → Query Rewriting → Query Decomposition / Keyword Extraction → Hybrid Search” 구조로 확장했고, 검색 정확도를 90% 이상 개선할 수 있었습니다. 약 1년간 서비스를 운영한 이후, 기존의 영어, 한국어 중심 구조에서 베트남어, 중국어, 일본어를 포함한 다국어 환경으로 확장되면서 기존 검색 방식을 재검토했습니다. 언어별로 관리해야 했고, 동시에 이를 실무자가 동적으로 관리할 수 있는 체계까지 함께 설계해야 했습니다. 이 기간동안 LLM의 성능이 빠르게 발전해 큰 context를 낮은 비용으로 활용할 수 있는 환경이 마련되었습니다. 이에 기존 벡터 중심 구조를 유지하는 대신, 검색 전략을 다시 설계했습니다. 문서를 대주제–중주제–소주제로 구조화하고, LLM이 직접 검색 tool을 호출해 필요한 영역을 동적으로 탐색하도록 하는 Agentic-RAG 방식을 도입했습니다. 그 결과 다국어 환경에서도 안정적인 검색 성능을 확보했으며, 벡터 인덱스 관리와 언어별 튜닝 비용을 줄이는 등 확장성과 운영 효율까지 개선할 수 있었습니다.

< 2.3 답변 생성 품질 고도화 > 검색 정확도를 확보한 이후에는 생성 단계의 품질을 어떻게 안정적으로 통제할 것인지에 집중했습니다. 상담 도메인에서는 단순 정보 전달이 아니라 톤, 표현 방식, 필수 안내 요소 등 여러 조건을 동시에 만족해야 했습니다. 그러나 “친절하게”, “간결하게”와 같은 추상적 지시만으로는 출력 편차를 줄이기 어려웠습니다. 이를 해결하기 위해 QnA 기반 few-shot 방식을 도입했습니다. 특히 친절도와 같은 정성 기준은 강도별 예시를 함께 제시해 출력 범위를 구체화했습니다. 또한 검색된 문서의 메타데이터를 활용해 가격 안내, 의료진 소개, 이벤트 안내 등 반드시 포함해야 하는 요소를 명시하고 이를 충족하는 예시를 제공해 출력을 통제했습니다. 그 결과 답변의 일관성과 조건 충족률을 개선했으며, 운영 과정에서 발생하던 표현 편차와 정보 누락 문제를 유의미하게 줄일 수 있었습니다.

< 2.4 모니터링 자동화 > 서비스 운영 과정에서 하루 200~300건의 대화를 수동으로 모니터링하는 일은 상당한 비용이 발생했습니다. 이를 해결하기 위해 LLM 기반 평가 자동화 파이프라인을 구축했습니다. RAG 구조의 특성을 반영해 두 가지 핵심 metric을 정의했습니다. 해당 질문에 대해 참조 가능한 충분한 문서가 존재하는지와 생성된 답변이 실제로 검색된 문서에 근거했는지입니다. 이를 활용해 각 대화에 대해 자동으로 점수를 산출하고, 일정 임계값 이하의 사례만 선별해 확인할 수 있게 되었습니다. 그 결과 전체 대화를 모니터링하던 방식에서 벗어나, 리스크가 높은 케이스 중심의 선택적 모니터링 체계로 전환할 수 있었습니다. 이를 통해 모니터링 비용을 유의미하게 줄일 수 있었습니다.

[3. 예측지표 기반 렌즈 삽입술 보조 서비스 개발 - ML 모델링 및 E2E 개발 역량] 렌즈 삽입술 의사결정을 보조하기 위해 핵심 예측 지표를 정의하고, 이를 기반으로 한 AI 서비스를 기획하고 개발했습니다. 현장 인터뷰를 진행하며 현장에서 실제로 활용되기 위해서는 예측 성능을 넘어, 결과에 대한 신뢰성, 변수–결과 관계의 해석 가능성, 의학적 직관과의 일관성, 그리고 실제 운영 환경에서의 안정성까지 함께 설계되어야 한다고 판단했습니다. 따라서 다음과 같이 프로젝트를 진행했습니다. 1. 사용자가 결과를 신뢰할 수 있는지에 대한 평가지표 설계 2. 설명력을 고려한 모델 설계 3. 데이터 수집–전처리–학습–추론–모니터링으로 이어지는 End-to-End ML system 구축

< 3.1 신뢰성을 고려한 평가지표 설계 > 현장의 인터뷰를 진행하며 깨달은 점은 예측 성능과 사용자가 느끼는 신뢰성은 반드시 비례하지 않는다는 것이었습니다. 예측지표 (MAE나 R²) 가 개선되더라도 결과가 의학적 직관과 어긋나는 순간, 모델에 대한 신뢰는 급격히 떨어졌습니다. 현장의 실무자들은 모델의 예측성능보다 모델의 예측결과가 공유하고 있는 직관과 일치하는지를 통해 모델의 신뢰성을 판단했습니다. 예를 들어 렌즈 사이즈가 커질수록 수정체와 렌즈 사이의 거리는 증가해야 한다는 것은 현장에서 공유되는 기본적인 직관입니다. 하지만 예측 지표만을 고려한 모델의 추론 결과는 이 직관과 일치하지 않을 수 있었습니다. 따럿 저는 예측지표 외에 신뢰성을 위한 평가지표를 설계했습니다. 독립변수와 종속변수 간의 관계가 의학적 직관을 얼마나 일관되게 반영하는지를 평가하는 지표를 추가했습니다. 이를 통해 모델의 신뢰성을 판단하고, 현장의 신뢰성을 높이기 위한 방향으로 고도화할 수 있었습니다.

< 3.2 설명력이 중요한 현장에 맞는 모델 설계 > 3.1에서 정리한 것처럼, 이 프로젝트에서 모델을 만들 때 중요한 것은 단순히 예측 오차를 줄이는 것이 아니었습니다. 현장에서 활용할 수 있으려면 “왜 이런 결과가 나왔는지” 설명할 수 있어야 했고, 예측값 자체뿐 아니라 모델이 발견한 패턴이 의학적으로 납득 가능해야 했습니다. 특히 현장에서는 블랙박스 모델에 대한 거부감이 존재했습니다. 예측 성능이 높더라도 변수와 결과 사이의 관계를 설명할 수 없다면 신뢰를 얻기 어렵다는 점을 인터뷰를 통해 확인할 수 있었습니다. 또한 모델이 학습한 패턴이 의학적 직관과 충돌할 경우, 평균 성능과 무관하게 서비스는 사용되지 않을 가능성이 높았습니다. 이에 저는 독립변수와 종속변수 간의 관계를 직접적으로 모델링하고, 이를 시각적으로 설명할 수 있는 Generalized Additive Model(GAM)을 선택했습니다. GAM은 각 변수의 효과를 분리해 해석할 수 있어, 변수 변화에 따른 예측값의 방향성과 형태를 명확히 제시할 수 있습니다. 이를 통해 모델의 설명력을 확보함과 동시에, 데이터에서 발견한 패턴이 의학적으로 타당한지 검증할 수 있는 구조를 만들었습니다. 또한 인터뷰를 통해 단순한 조건부 평균 예측만으로는 의사결정을 도울 수 없다는 것을 알 수 있었습니다. 수술의 평균적인 예측값 외에 상한과 하한에 대한 추정이 필요했고, 위험도를 정량화해서 제공해야 했습니다. 이를 위해 pinball loss를 통해 조건부 분위수를 추정했고, 추정된 분위수로 Metalog를 통해 분포를 추정했습니다. 이를 통해 특정 범위에 대한 확률을 구할 수 있었고, 위험도를 정량화해 제공할 수 있었습니다. 결과적으로 모델은 독립변수와 종속변수 간의 관계를 시각적으로 제공할 수 있었고, 조건부 분위수를 추정해 수술 결과의 분포와 위험도를 정량화해 제공할 수 있었습니다. 이를 통해 서비스는 현장의 신뢰를 얻고, 의사결정에 도움을 제공할 수 있었습니다.

< 3.3 End-To-End 아키텍처 설계 > 모델을 만드는 것보다 더 중요하게 생각한 것은, 이 모델이 실제 현장에서 안정적으로 작동할 수 있는 구조를 설계하는 일이었습니다. 데이터 흐름을 안정적으로 관리할 수 있는 데이터 파이프라인을 설계했습니다. 파편화되어 있던 검사결과 이미지 및 스프레드시트 데이터는 OCR 파이프라인과 정규화 전처리 파이프라인을 거쳐 데이터 웨어하우스에 적재되도록 구성했습니다. 이후 모델링·분석 목적의 데이터는 별도의 데이터 마트로 분리 구축해, 원천 데이터와 분석용 데이터를 명확히 분리했습니다. 또한 피처 스토어를 도입해 피처 정의, 변환 로직을 중앙에서 관리하고, 학습과 추론 시점에 동일한 피처가 생성되도록 설계했습니다. 이를 통해 학습–서빙 간 데이터 차이를 원천적으로 차단하고, 재현성과 운영 안정성을 확보했습니다. 모델을 만드는 실험을 진행하면서 가장 경계한 것은 데이터 누수였습니다. 테스트셋을 반복적으로 확인하는 행위 자체가 과적합으로 이어질 수 있기 때문입니다. 따라서 데이터를 날짜 기준으로 분리하고 서빙 직전 특정 시점에만 검증하도록 프로세스를 구축해 실험 성능과 실제 추론 성능이 일치할 수 있도록 했습니다. 모델 서빙은 BentoML 기반으로 구성했습니다. 경량화된 API 서버 형태로 패키징해 배포 일관성을 확보하고, 추후 모델 버전 관리와 롤백이 용이하도록 설계했습니다. 단순히 모델만 배포하는 것이 아니라, 백엔드 API와 프론트엔드 인터페이스까지 직접 구현해 예측 결과와 변수별 기여도를 함께 시각화함으로써 의료진이 결과를 이해하고 활용할 수 있도록 구성했습니다. 또한 운영 단계에서의 silent failure를 방지하기 위해 모니터링 파이프라인을 별도로 구축했습니다. 주기적으로 예측 분포, 주요 피처 분포, 성능 지표를 계산해 Slack으로 리포트를 자동 전송하도록 설계했으며, 이를 통해 데이터 드리프트나 성능 저하를 조기에 탐지할 수 있는 체계를 마련했습니다. 단순한 모델 개발이 아니라, 데이터 수집부터 서빙, 모니터링까지 이어지는 End-to-End ML 시스템을 책임지고 구축했습니다.

[4. 마무리 - 현장에서 작동하는 AI를 설계하는 사람] 제가 수행해온 프로젝트들은 공통적으로 “현장에서 실제로 작동하는 AI 시스템을 설계한다”는 목표를 중심에 두고 있습니다. 단순히 모델을 만드는게 아닌, 실제 현장에서 작동할 수 있는 시스템을 설계하고 구현해왔습니다. 이러한 경험을 바탕으로 FDDS로서 현장에서 실제 가치를 만들 수 있는 시스템을 구현하겠습니다.
