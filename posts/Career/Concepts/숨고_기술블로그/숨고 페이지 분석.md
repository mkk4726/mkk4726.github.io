# 숨고 페이지 분석

## 전제

이 문서는 **2026-04-26 기준으로 웹에서 관찰 가능한 흐름**과, 그로부터 추정한 내부 로직을 정리한 메모다.

- `관찰한 것`과 `추정한 것`을 분리해서 적는다.
- 실제 내부 로직은 다를 수 있다.
- 특히 요청서 작성 이후의 매칭/랭킹 방식은 외부에서 100% 확인할 수 없으므로, **도메인적 추론**으로만 적는다.

---

## 한 줄 요약

숨고는 단순 검색 결과 페이지에서 끝나는 구조라기보다, **서비스 선택 -> 구조화된 요청서 작성 -> 조건에 맞는 고수에게 견적 도달 -> 견적 비교/상담**으로 이어지는 `request-driven marketplace`에 가깝다.

즉, 검색은 끝이 아니라 **정교한 요청서 생성을 위한 첫 진입점**처럼 보인다.

---

## 1. 현재 보이는 고객 흐름

### 관찰 1. 홈에서 바로 검색창과 카테고리 진입을 강하게 유도한다

2026-04-26 기준 숨고 웹 홈에는 다음 요소가 보인다.

- 상단 주요 메뉴: `견적요청`, `고수찾기`, `마켓`, `커뮤니티`
- 메인 검색 입력: `어떤 서비스가 필요하세요?`
- 지역 표시: 예시로 `강남구`
- 카테고리 탐색: 이사/청소, 설치/수리, 인테리어, 외주, 취업/직무, 취미/자기계발 등
- 하단 추천 서비스 묶음

이 구조만 보면 숨고는 일반적인 "검색 엔진"처럼 보일 수 있다. 하지만 실제 전환 지점은 검색 결과 열람 자체보다 **견적 요청**에 더 가깝다.

### 관찰 2. 가격 페이지도 사실상 견적 요청 랜딩 페이지 역할을 한다

여러 `prices` 페이지를 보면 공통적으로 다음 흐름을 반복한다.

- 예상 견적 정보 제공
- 누적 견적요청 수, 활동 고수 수 노출
- `무료 견적 받기` CTA
- `숨고에서 ... 견적 받는 방법` 안내
  - 견적 요청하기
  - 견적 받기
  - 상담하기
  - 고용하기

즉 가격 콘텐츠 페이지도 단순 SEO 랜딩이 아니라, **사용자를 요청서 작성으로 보내는 퍼널 상단**으로 설계된 것처럼 보인다.

### 관찰 3. 사용자가 "견적서 받기"를 누르면 질문을 하나씩 선택하게 된다

직접 사용해본 흐름상, 사용자는 원하는 서비스를 고른 뒤 바로 고수 리스트만 받는 것이 아니라 **질문을 하나씩 답하는 요청서 플로우**로 진입한다.

이건 매우 중요한 포인트다.

- 사용자는 처음엔 "서비스 검색"처럼 들어오지만
- 실제 전환은 "구조화된 요청서 작성"으로 유도된다
- 즉 숨고의 핵심 입력값은 자유 텍스트 검색어보다 **요청서 응답값**일 가능성이 높다

---

## 2. 이 흐름이 의미하는 것

### 관찰 기반 해석

숨고는 고객이 원하는 것을 검색창 하나로 정확히 표현하기 어려운 문제를 안고 있다.

예를 들어:

- `청소`
- `영어 과외`
- `웹 디자인`
- `도배장판`

이런 서비스는 이름은 같아도 실제 요청 조건이 너무 다르다.

- 지역
- 예산
- 일정
- 범위
- 난이도
- 목적
- 원하는 방식

그래서 숨고는 검색 결과를 먼저 잘 보여주는 것보다, **질문을 통해 요청을 구조화하는 것**이 더 중요해 보인다.

즉 현재 UX는 "검색 -> 리스트 탐색"보다 **검색 -> 요청서 정제 -> 매칭**에 최적화된 구조에 가깝다.

---

## 3. 추정되는 내부 로직

아래는 외부 관찰과 공식 도움말을 바탕으로 한 추정이다.

### 추정 1. 서비스별로 정해진 요청서 스키마가 있다

질문을 하나씩 선택하게 한다는 건, 서비스마다 미리 설계된 요청서 항목이 있다는 뜻이다.

예상되는 구조:

- 서비스 ID
- 지역
- 세부 서비스 유형
- 일정 관련 항목
- 예산/규모 관련 항목
- 자유 입력 텍스트
- 기타 서비스별 특화 속성

즉 고객 요청은 단순 문장이 아니라, **정규화된 attribute bundle**로 저장될 가능성이 높다.

### 추정 2. 매칭 로직의 핵심은 "검색어"보다 "요청 조건 일치도"다

도움말에 따르면 요청서를 보내면 `조건에 맞는 여러 고수들을 매칭`해주고, 고수찾기에서 특정 고수에게 지정 요청을 보냈다가 응답하지 못하면 `조건이 비슷한 다른 고수들의 견적서`를 보내준다고 한다.

이 표현을 보면 숨고의 중심 로직은 전통적인 문서 검색보다 다음에 더 가깝다.

- 요청 조건과 고수 프로필/설정의 일치 여부 판단
- 특정 고수 실패 시 유사 조건의 대체 고수 탐색
- 고객 요청을 다수 고수에게 분배하는 방식의 매칭

즉 "검색 결과를 랭킹해서 보여준다"기보다, **요청서를 기준으로 고수 후보 집합을 만든 뒤 전달/노출**하는 구조일 가능성이 높다.

### 추정 3. 요청서 질문은 단순 UX가 아니라 매칭 정밀도를 올리기 위한 feature collection이다

고수 대상 도움말에 나온 `스마트견적` 설명을 보면,

- 설정한 조건과 `100% 일치하는 요청`에만 견적 발송
- `기존 대비 2배 늘어난 선택 항목`
- 최근 견적 발송 패턴을 분석해 `AI가 최적의 조건 추천`

이라는 문구가 있다.

이건 중요한 힌트다.

즉 고객이 답하는 각 질문은 단순한 폼 입력이 아니라:

- 고수의 타겟 조건과 매칭되는 feature
- 자동 견적 발송 조건의 필드
- 고용 가능성 예측에 들어갈 변수

일 가능성이 높다.

내 해석은 이렇다.

> 요청서 단계에서 질문을 많이 받는 이유는 UX를 귀찮게 만들기 위해서가 아니라, 이후 매칭/자동견적/견적 발송 효율을 높일 핵심 feature를 수집하기 위해서다.

### 추정 4. 숨고의 "검색"은 두 층으로 나뉘어 있을 수 있다

#### 1) 서비스 탐색 검색

- 사용자가 어떤 서비스를 찾는가
- 서비스 taxonomy 안에서 적절한 서비스로 보내는 단계
- 홈 검색창, 카테고리, 연관 서비스, 가격 페이지가 여기에 해당

#### 2) 고수 매칭 검색

- 구조화된 요청서를 바탕으로 어떤 고수에게 보낼지 결정하는 단계
- 내부적으로는 search보다 matching/filtering/scoring에 가까울 수 있음

즉 사용자가 느끼는 표면 UX는 "서비스 검색"이지만, 숨고의 실제 핵심 엔진은 **request-to-pro matching**일 가능성이 높다.

---

## 4. 왜 질문을 하나씩 받는가

내 현재 가설은 세 가지다.

### 1. 고객 의도를 더 정확한 서비스 단위로 좁히기 위해

처음 입력은 너무 넓다.

- "청소"만으로는 입주청소인지, 거주청소인지, 상가청소인지 모른다
- "과외"만으로는 대상, 과목, 수준, 목적이 없다

질문을 하나씩 던지면 고객 의도를 서비스 가능한 단위로 점점 구체화할 수 있다.

### 2. 고수 입장에서 "응답할 만한 요청"인지 판단 가능하게 만들기 위해

고수는 아무 요청에나 견적을 보내고 싶지 않다.

- 지역이 안 맞을 수 있음
- 일정이 안 맞을 수 있음
- 예산이 안 맞을 수 있음
- 작업 범위가 내 전문성과 안 맞을 수 있음

즉 질문은 고객만 위한 게 아니라, **고수의 응답 효율을 위한 정보 압축 장치**이기도 하다.

### 3. 이후 자동화와 AI 추천을 붙이기 쉬워지기 때문에

구조화된 필드가 많을수록 다음이 쉬워진다.

- 자동 견적 발송
- 유사 요청 추천
- 고용 가능성 높은 고수 우선 노출
- 고객에게 맞는 추가 질문 추천
- 가격 예측
- 요청서 작성 중 이탈 예측

즉 현재 질문 플로우는 단순 폼이 아니라, **플랫폼의 추천/매칭/자동화 기반 데이터 수집 장치**일 가능성이 크다.

## 5. 현재 페이지 구조에서 보이는 제품 철학

### 철학 1. "검색 결과 보여주기"보다 "견적 요청까지 보내기"가 더 중요하다

숨고의 많은 페이지가 결국 `무료 견적 받기`로 수렴한다.

이건 곧 성공 지표가 단순 페이지뷰나 검색 클릭률보다:

- 요청서 작성 완료율
- 요청당 견적 수신율
- 견적 이후 상담 전환율
- 최종 고용 전환율

같은 퍼널 지표에 더 가까울 가능성이 높다는 뜻이다.

### 철학 2. 범용 검색보다 카테고리별 구조화가 중요하다

숨고는 서비스 종류가 너무 많다. 그래서 통합 검색창 하나만으로 다 풀기보다, **카테고리별 질문 설계**가 매우 중요해 보인다.

즉 숨고의 경쟁력은 단순 검색 relevance보다:

- taxonomy 설계
- 요청서 설계
- 서비스별 핵심 attribute 정의
- attribute와 고수 특성의 매핑

이런 데 있을 수 있다.

### 철학 3. 양면시장 최적화 문제다

고객 입장에서는 빨리 정확한 고수를 찾고 싶고,
고수 입장에서는 성사 가능성이 높은 요청만 받고 싶다.

그래서 숨고의 질문 플로우는 고객 UX만 최적화하는 게 아니라, **양면시장 양쪽의 효율을 동시에 맞추는 장치**로 보인다.

## 6. 숨고 Data Scientist 관점에서 떠오르는 문제들

이 흐름을 보면 Data Scientist가 풀 수 있는 문제가 꽤 명확해진다.

### 1. 요청서 질문 최적화

- 어떤 질문을 먼저 물어야 이탈이 적은가
- 어떤 질문은 생략 가능하고, 어떤 질문은 꼭 받아야 하는가
- 서비스별로 질문 수와 전환율의 trade-off는 어떤가

### 2. 요청서 완성 이전 의도 추정

- 고객이 몇 개 질문만 답했을 때도 어떤 서비스/고수 후보가 유력한가
- 다음 질문을 무엇으로 던지면 정보 이득이 가장 큰가

### 3. 매칭 품질 개선

- 어떤 요청에 어떤 고수가 실제로 잘 고용되는가
- 단순 attribute match 말고 전환 가능성까지 반영할 수 있는가
- 고수 응답률과 고용률을 함께 보는 scoring이 가능한가

### 4. 받은 견적 랭킹

- 여러 견적 중 어떤 고수를 위에 보여줄 것인가
- 가격, 응답속도, 리뷰, 활동성, 유사 요청 성과를 어떻게 섞을 것인가

### 5. 자동 견적과 추천 고도화

- 스마트견적 추천 조건을 어떻게 만들 것인가
- 고수에게 어떤 요청을 자동 발송해야 효율이 가장 높은가

---

## 7. 지금 내 해석

현재 숨고를 겉으로만 보면 "서비스 검색 후 견적 받기"처럼 보이지만, 실제 핵심은 더 아래에 있다.

> 숨고는 검색 플랫폼이라기보다, 고객의 모호한 수요를 구조화된 요청서로 바꾸고, 이를 고수 매칭과 견적 흐름으로 연결하는 플랫폼이다.

따라서 숨고에서 말하는 검색 개선도 단순 웹검색 relevance 문제가 아니라, 아래 전체를 포함할 가능성이 있다.

- 어떤 서비스로 보내는가
- 어떤 질문을 더 받는가
- 어떤 고수에게 보내는가
- 어떤 견적을 먼저 보여주는가

즉 검색, 추천, 매칭, 요청서 설계가 사실상 분리되지 않은 하나의 문제처럼 얽혀 있을 수 있다.

## 8. 다음에 더 보면 좋을 것

- 실제 요청서 질문 흐름을 서비스 2~3개 더 비교해 보기
- `고수찾기`와 `견적요청` 흐름이 어디서 갈라지고 어디서 합쳐지는지 보기
- `빠른 상담이 가능한 고수` 노출 기준 추정
- 가격 페이지의 SEO 유입 -> 요청서 전환 설계 분석
- 받은 견적 화면에서 랭킹 기준 추정
