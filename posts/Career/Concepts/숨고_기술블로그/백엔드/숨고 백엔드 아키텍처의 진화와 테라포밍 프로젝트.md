# 숨고 백엔드 아키텍처의 진화와 테라포밍 프로젝트

## 원문

- 발행일: 2025.08.22
- 링크: https://soomgo.team/blog/posts/68a8406e0bad0c87176b8079

## 한 줄 요약

숨고는 PHP 모놀리식에서 Django 모놀리식, Flask 기반 MSA 확산기를 거쳐, 테라포밍 프로젝트를 통해 도메인 중심 MSA와 Kong 기반 게이트웨이 구조로 재정비했다.

## 구조 변화의 역사

### 1. 초기

- PHP 기반 모놀리식
- 문제:
  - 전체 배포 필요
  - 단일 장애가 전체 서비스에 영향
  - 유지보수 복잡도 증가

### 2. 다음 단계

- Python Django 기반 `Soomgo-web`
- 생산성과 안정성은 좋아졌지만 여전히 모놀리식 한계를 가짐

### 3. MSA 초기 확장기

- Flask 기반 경량 서비스들이 늘어나며 도메인 분리를 시작
- 하지만 서비스 수가 많아지면서 의존성, 운영 복잡도, 관리 난이도가 다시 문제로 떠오름

## 테라포밍 프로젝트의 핵심

### Soomgo-web 완전 제거

- 전체 API의 절반 가까이를 담당하던 `Soomgo-web`을 완전히 도메인 서비스로 이관
- 사실상 마지막 큰 모놀리식을 제거한 전환점

### 도메인 중심 재설계

- 서비스 단위가 아니라 `도메인 단위`로 재편
- 각 도메인이 API와 인프라를 독립적으로 소유
- 높은 응집도, 낮은 결합도를 구조 원칙으로 삼음

### 인프라 구조 개편

- Ingress 뒤에 Kong을 두는 구조로 단순화
- 공용 워커 노드 그룹 중심 구조에서 도메인별 워커 노드 그룹 분리로 이동
- 내부 통신은 gRPC와 REST를 혼용하되, 직접 호출 중심으로 정리

## 게이트웨이 개선

- 기존 Python API Gateway는 요청 조합과 응답 병합을 담당했다.
- 문제:
  - I/O 병목
  - 싱글스레드 기반 동시성 한계
  - 복잡도 증가
- 해결:
  - Python Gateway 제거
  - Kong API Gateway 도입

### 기대 효과

- 중간 계층 제거
- 네트워크 홉 감소
- 인증과 라우팅의 고성능 처리
- 서비스 간 데이터 흐름 단순화

## 내부 통신 개선

- REST 중심 통신에서 `gRPC`를 적극 도입
- 효과:
  - 바이너리 프로토콜 기반 빠른 전송
  - `proto` 기반 계약 명세
  - 높은 처리량과 낮은 지연시간

## 성과

- 대표 API에서 응답 시간이 대폭 줄었다.
- 예시:
  - `get_campaign_list`: 37,107ms -> 567ms
  - `get_products`: 54,480ms -> 550ms
  - `get_request_intro_data`: 9,015ms -> 465ms

## 인상적인 포인트

- 숨고는 `모놀리식 -> MSA`만 이야기한 게 아니라, `MSA 확산 후 다시 정리하는 단계`까지 다뤘다.
- 구조를 쪼갠 뒤 생긴 운영 복잡도까지 다시 정비했다는 점이 중요하다.
- 테라포밍이라는 표현이 어울릴 정도로, 단순 성능 튜닝이 아니라 `구조적 지형 자체를 바꾼 프로젝트`다.
