# FastAPI를 도입하면서

## 원문

- 원제: FastAPI를 도입하면서 : 코드 최적화부터 테스트까지
- 발행일: 2025.06.20
- 링크: https://soomgo.team/blog/posts/6854da54fdab889ca9498d24

## 한 줄 요약

숨고는 커뮤니티 프로젝트에 FastAPI를 도입해 개발 효율과 성능을 확보했지만, 비동기 전환과 Pydantic v2 마이그레이션, 테스트 최적화 같은 현실적인 숙제를 함께 풀어야 했다.

## 도입 이유

- 새로운 도메인 요구사항을 빠르게 반영할 필요가 있었다.
- FastAPI의 장점이 명확했다.
  - 의존성 주입
  - Pydantic 기반 검증
  - 자동 문서화
  - 비동기 처리 성능

## 실제 운영에서 맞닥뜨린 문제

### 비동기 전환

- SQLAlchemy 1.4의 비동기 방식은 기존 쿼리 패턴과 달랐다.
- MongoDB도 기존 `pymongo` 대신 비동기 라이브러리인 `motor`로 전환해야 했다.
- 초기에는 자체 ODM을 구현했고, 이후 `beanie`로 옮겨 생산성을 높였다.

### FastAPI 0.100 / Pydantic v2 마이그레이션

- FastAPI가 0.100에서 내부적으로 Pydantic v2를 채택했다.
- Rust 기반 재작성으로 성능은 좋아졌지만 호환성 이슈가 컸다.
- `bump-pydantic`을 활용했지만 복잡한 사용 사례가 많아 약 `3주`에 걸친 수작업 보완이 필요했다.

### 테스트 속도

- 테스트 실행 시간이 어느 시점에 약 `10분`까지 늘어났다.
- 해결:
  - `pytest-xdist`로 병렬 실행
  - 목 데이터 생성 과정 최적화
  - 병렬 환경에서의 간헐 실패 보완
- 결과적으로 `10분 -> 1분대`로 단축했다.

## 좋았던 점

- 타입 힌트와 Pydantic 덕분에 검증 로직이 단순해졌다.
- Background Tasks가 간단한 비동기 작업 처리에 유용했다.
- 공식 문서 품질이 좋아 초기 도입과 학습에 도움이 됐다.

## 아쉬웠던 점

- SQLAlchemy와 Pydantic 간 변환 비용이 꽤 있었다.
- 초기 프로젝트 구조는 확장성을 충분히 고려하지 못해 이후 여러 차례 손봐야 했다.

## 배운 점

- 기술 선택은 유행보다 `도메인 요구사항 적합성`이 중요하다.
- 높은 테스트 커버리지는 대규모 마이그레이션에서 안전장치 역할을 한다.
- 완벽한 시작보다 지속적인 개선이 더 중요하다는 메시지가 강하다.
