# RDB -> NoSQL로의 마이그레이션 도전기

## 원문

- 발행일: 2026.02.05
- 링크: https://soomgo.team/blog/posts/69847d109427f2d42068b8b4

## 한 줄 요약

숨고는 요청서 도메인의 복잡한 다중 조인 구조를 MongoDB 기반으로 재설계해, 응답 속도와 개발 생산성을 함께 끌어올렸다.

## 배경

- 요청서는 숨고 매칭과 매출 흐름을 떠받치는 핵심 도메인이다.
- 기존 MariaDB 구조에서는 요청서 한 건 조회에 `7~10개 테이블 조인`이 필요했다.
- 기능이 늘어날수록 컬럼과 테이블이 계속 늘어나 스키마 수정, 마이그레이션, 배포 부담이 커졌다.
- 결과적으로 `새로운 기능을 빠르게 실험하기 어려운 구조적 병목`이 됐다.

## 전환 전략

### 목표

- 조인 제거로 쿼리 복잡도 감소
- 스키마 유연화
- 데이터 구조 단순화
- 조회 속도 개선

### 설계 원칙

- `가능한 적은 컬렉션으로 관리한다`
- 최종적으로 요청서 정보를 크게 두 컬렉션으로 정리했다.
  - 요청서 본문 컬렉션: 변경 가능성이 낮은 정보
  - 요청서 메타데이터 컬렉션: 자주 바뀌는 정보

### 인덱스 전략

- 복합 인덱스: `(status, service_id, created_at)`
- 위치 기반 검색용 `2dsphere`
- 기간 조회용 인덱스
- 참조용 `request_id`, `user_id`

## 마이그레이션 방식

- RDB를 한동안 `진실의 원천`으로 유지하면서 MongoDB에 동기화했다.
- 이중 쓰기를 먼저 적용해 두 DB에 동시에 반영했다.
- 과거 데이터는 스크립트로 `4천만 건 이상` 이전했다.
- 대량 입력 속도를 위해 인덱스는 이관 후 지연 생성했다.
- 읽기 경로는 일부 API부터 점진적으로 MongoDB로 전환했다.

## 구현 중 부딪힌 문제

### ID 이중 관리

- 클라이언트는 기존 `request_id` 중심으로 동작했고, MongoDB는 `_id`를 사용했다.
- 이를 맞추기 위해 MySQL 쪽에 `request2_id`를 추가하고 양방향 참조 구조를 만들었다.

### 날짜 포맷 불일치

- MySQL은 로컬 타임존, MongoDB는 UTC를 사용해 `ended_at` 조회 결과가 어긋났다.
- 마이그레이션 시 UTC로 통일하고 애플리케이션도 UTC 기준으로 정리했다.

## 결과

- 요청서 관련 주요 API 평균 응답 속도 약 `90%` 개선
- `6~7개 테이블 조인`이 단일 컬렉션 조회 수준으로 단순화
- 스키마 변경 시 DB 마이그레이션 부담 감소
- 도메인 이해 난이도 감소로 협업 효율 개선

## 인상적인 포인트

- 단순 DB 교체가 아니라 `기능 실험 속도를 높이기 위한 구조 변경`이라는 점이 명확하다.
- 이중 쓰기와 점진적 읽기 전환으로 운영 리스크를 낮춘 방식이 현실적이다.
- "컬렉션 수를 최소화한다"는 원칙이 운영 복잡도를 줄이는 데 크게 작용했다.

## 남은 과제

- 아직 일부 레거시 API는 RDB를 거친다.
- MongoDB Atlas 모니터링 기반 자동 알람 강화가 필요하다.
- 최종적으로 RDB 동결과 테이블 제거 시나리오를 마무리해야 한다.
