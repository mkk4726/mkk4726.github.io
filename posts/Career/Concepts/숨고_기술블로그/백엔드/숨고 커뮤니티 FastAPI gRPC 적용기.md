# 숨고 커뮤니티 FastAPI gRPC 적용기

## 원문

- 발행일: 2024.12.17
- 링크: https://soomgo.team/blog/posts/675ffa8eda5f1f5a9dfbdfde

## 한 줄 요약

숨고는 커뮤니티 서비스와 고수 서비스 간 리뷰 데이터 동기화를 위해 REST나 메시지 큐 대신 `gRPC + FastAPI 비동기 서버` 조합을 택했다.

## 문제

- 리뷰 데이터를 커뮤니티에서 활용하려면, 서로 다른 서비스와 DB 스키마 사이에 일관된 동기화가 필요했다.
- 고수 서비스는 RDB, 커뮤니티 서비스는 MongoDB 기반이라 데이터 모델이 달랐다.
- 단순 이중 쓰기나 느슨한 비동기 전달은 데이터 불일치 가능성이 있었다.

## 왜 gRPC였나

- HTTP/2 기반 바이너리 전송으로 성능이 좋다.
- ProtoBuf 기반이라 타입 안전성이 높다.
- 내부 레퍼런스가 이미 있어 도입 부담이 낮았다.
- FastAPI의 asyncio 모델과 결합했을 때 동시성 처리에 유리했다.

## 구현 포인트

### 인터페이스 설계

- `.proto` 파일로 서비스와 메시지 형식을 정의했다.
- 데이터 구조가 복잡하고 일정이 촉박해, 일부 요청은 `dict -> string -> JSON -> Pydantic 검증` 흐름으로 처리했다.
- 완전한 protobuf 모델링보다 `기존 서비스 로직 재사용`을 우선한 선택이다.

### 서버 구현

- `grpc.aio.server` 기반 비동기 서버로 구성했다.
- `async_run`, `init_middlewares` 데코레이터를 통해 실행 로직과 모니터링/버그 추적 초기화를 재사용했다.

### 검증과 비즈니스 로직

- 수신 데이터는 Pydantic으로 검증했다.
- 실제 도메인 로직은 facade 패턴 뒤로 분리해 gRPC 핸들러는 얇게 유지했다.
- 로깅과 모니터링 데코레이터는 메타클래스로 자동 적용했다.

### 테스트와 운영

- `pytest`와 gRPC stub을 사용해 비동기 테스트를 작성했다.
- 응답 클래스 형태뿐 아니라 생성된 리뷰 ID까지 검증해 데이터 일관성을 확인했다.
- 배포 후 New Relic에서 CUD 요청 응답 시간을 관찰했고, 평균 `70ms` 수준을 유지했다고 한다.

## 인상적인 포인트

- "gRPC가 더 빠르다" 수준이 아니라 `데이터 일관성` 문제를 기준으로 통신 방식을 선택한 점이 좋다.
- protobuf를 이상적으로 밀어붙이지 않고, 일정과 기존 구조를 고려해 현실적으로 절충했다.
- FastAPI + gRPC + Pydantic + facade + 메타클래스 조합이 숨고식 백엔드 스타일을 잘 보여준다.
