# 효율적인 유지보수를 위한 요청서 개선하기

## 원문

- 발행일: 2024.08.01
- 링크: https://soomgo.team/blog/posts/669f2bd581774a166f079fe4

## 한 줄 요약

숨고는 ViewPager UI에 강하게 묶여 있던 요청서 작성 구조를 재설계해, 도메인 모델과 유효성 검증을 UI 밖으로 끌어내고 유지보수성과 확장성을 높였다.

## 문제

- 기존 요청서 작성은 특정 UI(ViewPager)에 종속되어 있었다.
- 그 결과:
  - 모델과 UI가 강하게 결합
  - 유효성 검증이 UI 컴포넌트 내부에 분산
  - 요청서 등록 로직도 화면 쪽에 스며듦
  - 테스트와 재사용이 어려움

## 접근 방식

### 도메인 분석

- 요청서 작성 시나리오를 다시 정리했다.
  - 서비스 선택
  - 폼 작성
  - 사용자 인증
  - 요청서 등록
- 도메인 전문가와의 대화를 통해 핵심 모델을 다시 추출했다.
  - 요청서 폼
  - 질문
  - 질문 유형
  - 답변 명세
  - 답변
  - 유효한 답변
  - 요청서 초안
  - 사용자 인증 절차

### 핵심 재정의

- `유효한 답변`이라는 개념을 별도 모델로 식별했다.
- 인증 절차는 요청서 등록 단계에서만 필요하다고 분리했다.
- 기존 `BaseFormItem`, `QuestionFormItem`, `RegeditFormItem` 같은 UI 중심 용어를 도메인 언어로 정리했다.

## 구현 포인트

### 유효성 검증을 도메인 레이어로 이동

- 답변 검증 로직을 UI에서 분리했다.
- 질문 타입별 규칙을 관리하고, 룰 실행기로 답변의 유효성을 판별한다.
- 유효한 답변은 생성기를 통해서만 만들어지도록 설계했다.
- `eslint-plugin-import-access`를 활용해 생성 경로를 제한한 점도 흥미롭다.

### 요청서 등록을 Application Layer Service로 이동

- UI가 아니라 애플리케이션 레이어가 요청서 등록 흐름을 책임지게 했다.
- 덕분에 비즈니스 로직 독립성과 테스트 용이성을 확보했다.

## 결과

- 요청서 작성 프로세스가 더 유연해졌다.
- UI 요구사항이 바뀌어도 도메인 모델을 재사용하기 쉬워졌다.
- 유효성 검증 로직 재사용성이 높아졌다.
- 테스트 가능성과 유지보수성이 개선됐다.

## 인상적인 포인트

- 이 글은 단순 UI 리팩터링이 아니라 `도메인 재발견`에 가깝다.
- "유효한 답변"을 타입과 생성 규칙으로 강제한 접근이 깔끔하다.
- Mobile 글이지만 핵심 메시지는 DDD, 클린 아키텍처, 애플리케이션 서비스 설계다.
