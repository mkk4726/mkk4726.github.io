# 모바일 앱 매주 배포하기 규칙 정하기

## 원문

- 원제: 모바일 앱 매주 배포하기: 규칙 정하기
- 발행일: 2021.04.18
- 링크: https://soomgo.team/blog/posts/6673bb8d52107866fb86a79f

## 한 줄 요약

숨고 모바일 챕터는 매주 수요일 배포를 안정적으로 지속하기 위해, 스쿼드 단위 브랜치 전략과 단순한 버저닝 규칙을 명시적으로 정했다.

## 배포 프로세스

- 금요일까지 스쿼드별 자체 QA와 병합 준비
- 월, 화 통합 QA
- 수요일 배포

즉, `주간 배포 리듬`이 먼저 있고 그 리듬에 맞춰 브랜치 전략이 설계됐다.

## 왜 기본 Git flow를 그대로 쓰지 않았나

- 모바일은 여러 버전을 동시에 다뤄야 해서 GitHub flow보다 Git flow가 더 잘 맞았다.
- 하지만 Git flow만으로는 `특정 스쿼드 작업만 배포 제외`하는 상황을 다루기 어려웠다.

## 선택한 방식

- `develop` 공용 브랜치를 없앴다.
- 대신 `squad/{스쿼드명}` 형태의 스쿼드별 develop 브랜치를 사용했다.
- 메인 release 브랜치와 스쿼드별 release 브랜치를 함께 운용했다.

## 이 구조의 장점

- 통합 QA에서 특정 스쿼드 작업이 문제를 일으켜도,
- 메인 release 브랜치를 다시 만들고 나머지 스쿼드 작업만 병합해 배포를 이어갈 수 있다.
- 즉, 전체 배포가 한 팀의 문제로 멈추지 않게 설계한 것이다.

## 버저닝 규칙

- 정기 배포: `minor` 증가
- 핫픽스 배포: `patch` 증가

단순하지만 이 규칙 덕분에 다음 배포 버전을 예측하기 쉽고, 자동화 구현에도 유리했다.

## 인상적인 포인트

- 이 글은 브랜치 전략 자체보다 `배포 리듬을 조직적으로 유지하는 방법`에 가깝다.
- 숨고는 배포를 기술 이벤트가 아니라 여러 스쿼드의 협업 프로세스로 보고 있다.
