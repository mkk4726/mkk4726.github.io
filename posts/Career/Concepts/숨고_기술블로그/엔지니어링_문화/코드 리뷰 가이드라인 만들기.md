# 코드 리뷰 가이드라인 만들기

## 원문

- 발행일: 2021.12.14
- 링크: https://soomgo.team/blog/posts/6673bb8b52107866fb86a77d

## 한 줄 요약

숨고는 코드 리뷰를 단순 품질 검사보다 `사람 사이의 더 나은 대화`로 보고, 리뷰 문화에 대한 명시적 가이드라인을 만들었다.

## 왜 중요했나

- 모든 코드 변경은 리뷰를 거쳐야 제품에 포함된다.
- 팀이 커질수록 코드 리뷰에 대한 기준이 사람마다 달라질 수 있다.
- 그래서 모바일 챕터는 각자의 시행착오를 바탕으로 공통 원칙을 정리했다.

## 핵심 원칙

### 피드백 방식

- 가급적 제안과 함께 피드백하기
- 결정하지 말고 제안하기
- 개인의 관점에서 의견을 표현하기
- 필요하면 칭찬도 명시적으로 하기
- OIR 구조를 활용하기
  - Observation
  - Impact
  - Request

### 작성자와 리뷰어의 태도

- 적용 여부 판단은 PR 작성자의 몫
- 짧은 코드보다 이해하기 쉬운 코드 우선
- 합의한 컨벤션 따르기
- 리뷰하는 코드를 내 스타일로 만들려 하지 않기
- 내가 실천하지 않는 것을 강요하지 않기

### 협업 관점

- 작은 단위로 PR 만들기
- 겸손하기
- 코드와 자신을 동일시하지 않기
- 우리는 모두 같은 편이라는 점 기억하기
- 이케아 효과를 경계하기

## 인상적인 포인트

- 기술적 디테일보다 `관계와 대화 방식`을 규정하는 가이드라는 점이 좋다.
- "Bad / Good" 예시처럼 표현 방식까지 다루는 점이 실용적이다.
- 숨고는 리뷰를 코드 품질 활동이면서 동시에 팀 문화 장치로 보고 있다.
