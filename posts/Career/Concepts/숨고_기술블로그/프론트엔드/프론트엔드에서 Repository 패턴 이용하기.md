# 프론트엔드에서 Repository 패턴 이용하기

## 원문

- 발행일: 2021.12.07
- 링크: https://soomgo.team/blog/posts/6673bb8c52107866fb86a78f

## 한 줄 요약

숨고 모바일 챕터는 프론트엔드에서도 데이터 조회 기술보다 도메인 모델에 집중하기 위해 `Repository 패턴`을 사용한다.

## 문제의식

- 애플리케이션 레이어가 데이터 소스에 직접 접근하면
  - 쿼리 로직이 도메인 밖으로 새고
  - 모델이 비대해지고
  - 엔티티가 단순 데이터 컨테이너로 전락하기 쉽다.

## 핵심 원칙

- 애플리케이션 레이어에서 데이터 소스 접근 메커니즘을 감춘다.
- Aggregate 내부 객체는 루트 객체를 통해서만 접근하게 제한한다.

이 두 가지 제약이 있어야 모델 외적인 관심사를 덜 신경 쓰게 된다는 주장이다.

## Repository 패턴의 역할

- 특정 타입의 객체들을 하나의 개념적 집합처럼 다룬다.
- 어플리케이션 레이어는 질의 메서드로 필요한 객체를 요청한다.
- Repository는 내부에서 데이터베이스/API/시스템 API 접근을 캡슐화한다.
- 객체뿐 아니라 요약 계산도 반환할 수 있다.

## 얻는 이점

- 의도를 드러내는 단순한 인터페이스
- 도메인 모델 중심의 질의
- 저장소 기술과 도메인 설계의 분리
- 테스트용 가짜 구현 작성 용이

## 인상적인 포인트

- 모바일/프론트엔드 글이지만 사실상 DDD 기본기를 잘 설명하는 글이다.
- 숨고는 프론트엔드도 단순 UI 레이어가 아니라 `도메인 모델을 다루는 애플리케이션`으로 바라본다는 점이 드러난다.
