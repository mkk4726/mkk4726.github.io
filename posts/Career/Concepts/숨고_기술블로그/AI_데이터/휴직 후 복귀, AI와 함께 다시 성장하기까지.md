# 휴직 후 복귀, AI와 함께 다시 성장하기까지

## 원문

- 발행일: 2026.01.20
- 링크: https://soomgo.team/blog/posts/696f1630b65f9b5a859cd51d

## 한 줄 요약

숨고는 AI를 코드 자동 생성기가 아니라 `문제를 구조화하고 초안을 빠르게 만드는 동료`로 활용하는 문화를 갖추고 있다.

## 글의 핵심

- 복귀 후 가장 크게 체감한 변화는 `AI를 쓰느냐`보다 `어떻게 쓰느냐`였다.
- 팀은 AI에 문제를 맡기는 것이 아니라, 문제를 더 넓게 보고 더 빠르게 시작하기 위한 도구로 사용한다.
- 최종 판단과 품질 기준은 여전히 사람이 책임진다.

## 실무에서의 활용 방식

### 로그 분석

- 복잡한 스택트레이스를 AI에게 전달해
  - 에러 발생 지점
  - 호출 관계
  - 재현 조건
  - 의심 라이브러리/버전 이슈
  - 해결 전략 후보
  를 빠르게 요약받는다.
- 이 과정에서 원인 후보를 좁히는 속도가 기존 대비 `3~5배` 개선됐다고 설명한다.

### 테스트 코드 초안

- 사람은 먼저 엔드포인트, 기대 동작, 제약 조건을 명확히 정의한다.
- AI는 그 조건을 바탕으로 `setUp / tearDown / 기본 시나리오` 수준의 테스트 뼈대를 생성한다.
- 개발자는 여기에 인증, 멀티 테넌시, 응답 스키마 검증, 로깅/트레이싱 같은 실제 서비스 제약을 덧입힌다.

### 설계 리뷰 보조

- 회원 탈퇴 API 예시처럼, 기능 설계 단계에서
  - 트랜잭션 경계
  - 외부 연동 실패 시 보상/재시도
  - 감사 로그
  - 삭제 vs 비활성화 전략
  같은 사각지대를 먼저 점검하는 데 AI를 사용한다.

## 이 글에서 강조하는 원칙

### 1. AI는 탐색과 초안 작성에 강하다

- 로그 분석
- 리팩토링 분리 포인트 제안
- 테스트 스캐폴딩

이 영역에서는 생산성 효과가 크다.

### 2. 최종 품질은 사람이 보장한다

- 트랜잭션 안정성
- 동시성/경합
- 외부 시스템 장애 복구
- 도메인 규칙과 정책

이런 부분은 여전히 개발자의 검토가 필요하다.

### 3. 프롬프트 엔지니어링은 개발 역량이다

- 어떤 환경인지
- 어떤 제약이 있는지
- 무엇을 검증하고 싶은지

를 명확히 쓸수록 AI 답변의 품질이 올라간다.

## 인상적인 포인트

- 이 글은 "AI로 얼마나 빨라졌는가"보다 `개발자의 사고 방식이 어떻게 바뀌는가`에 초점을 둔다.
- 숨고의 AI 활용은 자동화보다 `설계 리뷰와 탐색 비용 절감`에 더 가깝다.
- 반복 작업은 AI에게 넘기고, 사람은 설계·리뷰·커뮤니케이션에 더 많은 시간을 쓰는 방향이 명확하다.
