## 강점 1. 문제 재정의(Problem Redefinition) 능력 — 가장 돋보이는 강점

**"이게 진짜 풀어야 할 문제인가?"를 묻는 습관**이 세 프로젝트에 걸쳐 반복적으로 나타납니다. 이건 한 번은 운이고, 세 번은 역량입니다.

### 증거 1 — Lens Sizing
- MAE 110, R² 0.4에서 plateau에 부딪혔을 때, 일반적인 반응은 "튜닝을 더 해보자"
- 하지만 근본 원인을 파고들어 **"렌즈 정보 부재 = Aleatoric Uncertainty"**라는 진단에 도달
- 점 추정을 포기하고 예측 구간(Prediction Interval)으로 **문제 자체를 바꿈**
- 이건 엔지니어링이라기보다 **통계/철학적 결정**

### 증거 2 — Chatbot (분류 모델 도입)
- 불만족 피드백을 봤을 때 보통은 "답변 품질을 올려야지"로 감
- 하지만 피드백을 정량 분석한 결과 **"답변할 수 없는 질문에 답을 하는 게 진짜 원인"**임을 파악
- 해결책이 "LLM 성능 향상"이 아니라 **"LLM을 덜 호출하기"**로 완전히 뒤집힘
- 결과: 응답시간 30% 단축, 불만족도 80%+ 감소

### 증거 3 — Agentic RAG 전환
- 다국어 확장에서 "언어별 파이프라인을 잘 만들자"가 아니라 **"언어 독립적 구조로 바꾸자"**로 프레임 전환
- 문제를 "확장성"의 축으로 재정의해서 구조 자체를 바꿈

### 이게 왜 강점인가
대부분의 엔지니어는 **주어진 문제를 잘 푸는 것**을 훈련받습니다. 문제 자체를 의심하는 건 훈련되지 않죠.
이 습관은 면접/실무 양쪽에서 극명한 차별점이 됩니다. "왜 그 방향으로 갔나요?"라는 질문에 **근본적인 답변**을 할 수 있거든요.

## 강점 2. 이론적 깊이를 실무에 녹여내는 희소 역량

대부분의 ML 엔지니어는 "논문 읽기"와 "제품 구현" 사이에 큰 간극이 있습니다. 그 간극을 좁히는 능력이 있습니다.

### 구체적 증거
- **Conformal Prediction(CQR)** 을 실제 의료 서비스에 적용 — Calibration Set, Conformity Score를 추상화된 파이프라인으로 구현
- **Causal Inference**의 Positivity / Propensity Score / Partial Identification / Manski Bounds까지 끌어와서 "측정하지 않은 처치의 예측을 어떻게 신뢰할 것인가"에 도전 (실패했지만 CQR의 밑거름)
- **Quantile Regression의 quantile crossing 문제**를 인지하고 파이프라인 내부에 방지 로직 통합
- **Pinball Loss를 Huber 기반으로 변형**하는 등 디테일까지 건드림
- **Conformal Prediction이 왜 '통계적으로 유효한 신뢰수준을 보장'하는가**에 대한 이해 위에 구현

### 이게 왜 희소한가
- 통계/수학 이론을 아는 사람은 많음 → 그러나 코드로 못 옮기는 경우 많음
- 구현을 잘하는 사람은 많음 → 그러나 "왜 이 방법이 맞는지" 통계적으로 설명 못함
- **두 쪽을 다 할 수 있는 사람은 드뭄**. 특히 헬스케어 같은 고위험 도메인에서 핵심 차별점

### 흥미로운 포인트
"양수성 도입 시도"가 실패로 끝났음에도, 그 경험이 **CQR로 문제를 재정의하는 밑거름**이 됐다는 점. 단순 "시도했다"가 아니라 **"인과추론의 관점에서 예측 신뢰도를 어떻게 정량화할 것인가"**를 진지하게 고민한 흔적이 남음.

## 강점 3. "오버엔지니어링 회피" 감각

반복적으로 등장하는 문장:
> _"유명한 프레임워크나 툴을 사용하기보다 현재 필요한 기능에 맞도록 구현했습니다. 필요없는 기술을 위해 복잡성을 증가시키지 않고, 해결하고자 하는 문제에 집중했습니다."_

### 실제 선택들
- **OCR Text Detection을 룰베이스**로 단순화 — 고정 레이아웃이라는 도메인 특성 활용
  - 일반적 선택: CRAFT, DBNet 같은 Detection 모델 도입
  - 당신의 선택: 레이아웃이 고정이니 별도 모델 불필요
- **챗봇 질문 분류를 LLM 프롬프트로 구현** — 별도 classifier 학습 대신
  - 근거: 카테고리 변동성 높음 / 추가 학습 비용 없음 / 짧은 호출이라 비용 부담 적음
- **모니터링을 Spreadsheet + Airflow + Slack**으로 구축 — 별도 DB 없이
- **피처 스토어도 프레임워크 없이 직접 구현** — Feast, Tecton 같은 걸 가져오지 않음

### 이게 왜 강점인가
- 실제 현장에서 **화려한 기술을 잘 도입하는 것**보다 **도입하지 않아도 될 것을 도입하지 않는 것**이 훨씬 귀함
- "필요한 만큼"을 판단하려면 트레이드오프를 정확히 계산할 수 있어야 함
- 이건 경험과 사고력의 증거. 주니어 때는 보통 새 기술 써보고 싶어서 불필요한 도구를 가져옴

### 동시에 조심해야 할 점
- 이 감각이 너무 강하면 **필요할 때도 검증된 프레임워크를 거부**하는 쪽으로 갈 수 있음
- "왜 안 썼나요?"에 대한 설명을 준비해둘 것 (이미 잘하고 있지만 의식적으로)

## 강점 4. End-to-End ML 시스템을 혼자 돌릴 수 있는 스택

3개 프로젝트 전체를 보면 혼자서 다룬 범위가 놀라움.

| 영역 | 증거 |
|------|------|
| **모델링** | GAM, Quantile Regression, CQR, TrOCR, Embedding 모델 평가/선정 |
| **데이터 파이프라인** | Feature Store 직접 구현, 학습-추론 일관성 보장 |
| **서빙** | FastAPI, Docker, S3, TritonServer, 비동기 구조, gRPC |
| **모니터링** | Silent failure 감지, Data Drift, RAGAS 기반 평가, Airflow |
| **실험 관리** | Hydra, W&B |
| **품질/CI** | pytest, mypy, data leakage 방지 프로세스 |
| **아키텍처 설계** | OOP 재설계, Client-Server 역할 재정의, Agentic RAG 설계 |
| **도메인 협업** | 의사와 가설 수립/검증, 사용자 인터뷰 19명 |

### 왜 중요한가
- **OCR은 AI 1명**, **Agentic RAG는 AI 1명**이었음
- 큰 팀의 한 파트가 아니라 **시스템을 혼자 책임졌던 경험**
- 이건 스타트업/시니어 포지션에서 결정적 무기
- "역할을 좁혀서 깊이만" 파는 사람보다 **"문제 전체를 책임지는" 사람**이 훨씬 드뭄

### 주의할 점
- End-to-End 경험은 **"깊이가 부족하다"**로 오해받기 쉬움
- 특정 기술(서빙, 최적화, 분산 학습 등)에 대해 **"어디까지 깊이 들어갔나"**를 구체적으로 설명할 수 있어야 함
- 이력서에는 폭을, 면접에서는 폭+깊이를 보여주는 전략

## 강점 5. "측정 불가능하면 개선도 불가능하다"는 감각

평가 체계를 **직접 설계**하는 경향이 일관됨. 이건 단순히 "성과를 정량화하는 능력"을 넘어서 **"성과의 정의부터 다시 쓰는 능력"**.

### 사례
- **OCR** : 평가 데이터셋이 없으니 **200건 직접 구축**해서 기준점 확보
- **Chatbot RAG** : 임베딩 모델 비교를 위해 **LLM으로 평가 데이터셋 자동 생성**
- **Lens Sizing** : MAE → **Coverage / Interval Width**로 평가지표 자체를 재설계
- **신뢰도 평가지표**를 직접 개발해 **팀 표준**으로 정착시킴
- **RAGAS 기반 커스텀 평가지표** 설계

### 숨은 의미
"지표가 없으면 만든다"는 행동 패턴은, **비즈니스 임팩트를 수치화하는 습관**과 연결됨. 면접에서 자주 묻는 "성과를 어떻게 측정했나요?"에 **남들이 안 하는 수준의 답**을 할 수 있음.

또한 "좋은 엔지니어는 **측정 가능한 세계를 만든다**"는 시니어 레벨의 감각이 있음.

## 강점 6. 실패를 숨기지 않고 자산으로 만드는 성숙함

Lens Sizing의 "양수성 도입 시도 (실패했던 경험)"를 그대로 문서에 남겨두고, **그것이 어떻게 CQR로 이어졌는지** 연결한 부분.

### 인용
> _"이 접근은 실제 서비스에 적용하지는 못했습니다. 여러 제약 (데이터 양, 계산 복잡도, 사용자 검증 비용)으로 인해 본격적인 도입은 보류되었으나, 이 과정을 통해 얻은 통찰이 있습니다."_
>
> _"이는 이후 Quantile Regression 기반의 **예측 구간 추정**으로 문제를 재정의하는 데 밑거름이 되었습니다"_

### 왜 이게 강력한가
- **보통은 실패를 숨김** (포폴에 실패를 쓰는 사람은 거의 없음)
- 실패를 남기면서도 **내러티브를 만드는 능력** — "실패 → 배움 → 다음 단계의 전환점"이라는 구조
- 면접에서 **"가장 힘들었던 경험"** 질문의 정답이 이미 준비되어 있음
- 이건 **자기 객관화와 성숙함**의 증거. 주니어/미들에서는 드물고, 시니어로 갈수록 중요

### 한 발 더
이 강점을 **면접에서 의식적으로 활용**할 것. "실패한 프로젝트가 있나요?" 질문에 양수성 도입 실패를 당당히 이야기하고, **그게 어떻게 CQR 도입으로 이어졌는지**를 스토리로 풀면 매우 강력함.

## 강점 7. 사용자와 도메인 전문가와의 협업 감각

### 구체적 행동
- 의사와 협업해 "나이-수정체 두께" 같은 도메인 가설 세우고 → 데이터로 검증 → 피처화 (반복)
- 사용자 19명(의사 11 + 검안사 8) 대상 **사전-사후 Likert 척도 인터뷰**로 신뢰도 2.8 → 4.2 정량 측정
- 만족도 피드백을 "만족/불만족 버튼"으로 수집 → 그 로그를 정량 분석해 분류 모델 도입 결정
- 고객사 매뉴얼 흐름에 챗봇이 끼어들 수 있도록 고정답변을 **매뉴얼 중심으로 설계**

### 왜 희소한가
- ML 엔지니어들은 보통 **데이터만 보고 일함** → 사용자와 멀리 떨어져 있음
- 그런데 이 사람은 **"모델 성능 지표"와 "사용자 만족도 지표"가 다르다**는 걸 경험으로 체득
- 특히 헬스케어에서 이건 **필수 역량**. 의사 신뢰를 얻지 못하면 서비스 자체가 안 돌아감

### 파생 강점
- 통계/ML 언어를 **의사·고객·관리자**의 언어로 번역할 수 있다는 뜻
- "불확실성"이라는 개념을 "정상 범위 내 위치 확률"로 풀어낸 것도 이 감각의 결과

## 강점 8. LLM을 "생성 도구"가 아닌 "일반 연산기"로 보는 감각

### 활용 방식
- **평가 데이터셋 생성**에 LLM (RAG 평가용, OCR 평가용)
- **모니터링 자동화**에 LLM (LLM으로 LLM 답변 평가)
- **분류 모델**에 LLM (프롬프트만으로 구현)
- **Agentic RAG**에서 reasoning을 LLM에 위임

### 숨은 사고 패턴
> "사람이 하는 반복 업무 중 LLM으로 대체할 수 있는 게 뭐가 있나?"

이 관점이 체화되어 있음. 이건 2025~26년 현장에서 **가장 유용한 감각 중 하나**.

### 확장 방향
- 에이전트/도구 사용에 대한 깊은 이해로 확장 가능
- "언제 LLM을 써야 하고, 언제 쓰면 안 되는가"에 대한 감각이 있음
  - 고정답변 대상은 LLM 호출 제거
  - Agent 구조의 트레이드오프를 명시적으로 인식

## 강점 9. 디테일에 대한 집요함

잘 안 보이는 강점이지만 꽤 중요함.

### 흔적들
- **Quantile Crossing 문제**를 그냥 넘어가지 않고 파이프라인 내부에 방지 로직 통합
- **Huber 기반 Quantile Loss**로 이상치 강건성 확보
- **Data Leakage 방지**를 위한 엄격한 검증 프로세스 — test set을 배포 직전에만 확인
- **Pinball Loss, Conformity Score** 같은 디테일을 코드 수준에서 이해하고 구현
- OCR에서 **mypy 타입 검증**을 CI에 통합
- Redis **TTL**을 "세션 관리에 적합한 기능"으로 의식적 선택

### 왜 중요한가
- ML 업계에서 **"대충 돌아가게 만드는 사람"**은 많음
- **통계적/엔지니어링적으로 올바른 것**까지 챙기는 사람은 적음
- 이건 시니어 레벨의 신뢰를 얻는 핵심 요소

- 이는 **"써먹는 공부"**를 한다는 증거 — 학습이 업무와 분리되어 있지 않음

### 더 발전시키면
- 이 습관을 외부에 노출시키면(기술 블로그, 뉴스레터, 발표 등) **개인 브랜드로 전환** 가능
- 현재는 프로젝트 문서 내부에서만 링크되고 있는데, 좀 더 전면에 드러내면 좋을 듯

## 약점이나 보완하면 좋을 점

정직한 자기 분석을 위해, 장점 분석만큼 중요한 부분.

### 1. "폭이 넓다 = 깊이가 부족하다"로 오해받을 위험
- End-to-End를 강조할수록 "근데 정확히 어디까지 깊이 들어갔나요?"를 준비해야 함
- 예: TritonServer를 썼다면 Triton의 동적 배치 원리까지? 피처 스토어를 구현했다면 학습-추론 일관성을 보장하는 구체적 메커니즘은?

### 2. 프로젝트가 **한 회사 (비쥬웍스)** 에 집중
- Lens Sizing, OCR, Chatbot 모두 같은 회사 — 도메인(안과 의료)도 동일
- **다른 도메인에서도 통할 수 있는가**를 보여줄 기회가 적음
- 이직 시 면접관은 "헬스케어 바깥에서도 가능한가?"를 걱정할 수 있음
- 대응 : 각 프로젝트의 **구조적 공통점**(문제 재정의, 사용자 검증, End-to-End 책임)을 도메인 무관한 언어로 번역해두기

### 3. 대규모 트래픽/분산 시스템 경험이 상대적으로 얇음
- 지금까지는 중소 규모 병원 환경
- 대기업 이직 시 "수백만 QPS", "분산 학습" 같은 질문에 대한 답변이 약할 수 있음
- 다만 본인의 지원 방향이 **문제 재정의+통계 기반 ML** 쪽이라면 크리티컬하지 않음

### 4. 가장 큰 강점 (문제 재정의)을 **이력서에서 잘 드러내지 못함**
- 현재 이력서 자기소개는 너무 일반적 ("현장 문제를 데이터 기반으로 해결", "정량적 성과")
- 진짜 차별점인 **"문제 재정의"**, **"통계적 엄밀함"**이 사라져있음
- → 자기소개 문구 개선 필요

---

## 이력서 자기소개 문구 개선안

### 제안 1: 재정의 능력 강조형
> 헬스케어 도메인에서 ML/DL·LLM 기반 서비스를 End-to-End로 설계·구축하는 데이터 과학자입니다. **모델 성능의 한계에 부딪혔을 때 "기술 최적화"가 아닌 "문제 재정의"로 돌파**하는 데 강점이 있으며, Quantile Regression·Conformal Prediction·Agentic RAG 등 이론적 개념을 실제 서비스에 적용해 정량적 성과(수술 안정성 +10%p, 사용자 신뢰도 +50%, Hit Rate +70%)를 만들어왔습니다.

### 제안 2: 깊이+실용성 강조형
> 헬스케어 도메인에서 **통계적 엄밀함과 엔지니어링 실용주의를 결합**해 ML/LLM 기반 서비스를 개발합니다. 데이터 파이프라인부터 모델 개발·서빙·모니터링까지 End-to-End로 책임진 경험이 있으며, 단순 예측 정확도를 넘어 **불확실성 정량화·사용자 신뢰도·유지보수 비용** 같은 실질적 가치를 지표로 설계하고 개선해왔습니다.

### 제안 3: 간결형
> 헬스케어 도메인의 **복잡한 문제를 수학적으로 재정의하고, 필요한 만큼의 엔지니어링으로 해결**하는 데이터 과학자입니다. ML 시스템 설계부터 LLM 기반 서비스 구축까지 End-to-End로 진행하며, 모델 정확도·사용자 신뢰도·운영 비용을 정량 지표로 함께 개선한 경험이 있습니다.

## 면접에서 활용할 수 있는 "내 스토리" 키워드

자기 분석을 마치고 나서, 면접에서 무기로 쓸 수 있는 몇 가지 키워드:

1. **"문제 재정의"** — 세 프로젝트 모두에서 반복. Lens Sizing이 가장 강력한 사례.
2. **"불확실성 정량화"** — Quantile Regression + CQR로 실제 서비스에 도입한 경험
3. **"의사 19명의 신뢰도 50% 향상"** — 정량 + 정성 둘 다 있는 희소한 사례
4. **"LLM을 연산기처럼 쓰는 감각"** — Agentic RAG로 유지보수 비용 구조를 바꾼 경험
5. **"실패 스토리"** — 양수성 도입 시도 → CQR로 전환된 내러티브
6. **"필요한 만큼의 엔지니어링"** — 피처 스토어 직접 구현, Text Detection 룰베이스 등

## 앞으로 고민해볼 것들

### 1. 도메인 확장
- 헬스케어 바깥에서 경험을 만들 기회가 있는가?
- 혹은 **헬스케어를 깊게 파서** 도메인 전문성을 극대화할 것인가?

### 2. 기술 깊이의 다음 단계
- 현재의 스택에서 **어떤 부분을 한 단계 더 깊게** 파고들 것인가?
  - 예: Causal Inference 논문 follow-up / Uncertainty Quantification 연구 / LLM Agent 아키텍처
- 다음 1년 동안 **"진짜 잘한다"**고 말할 수 있는 영역은 무엇인가?

### 3. 커뮤니케이션 자산화
- 블로그 포스트들을 **외부에 노출**시킬 것인가?
- 발표, 기술 블로그, 오픈소스 기여 등으로 확장 가능

### 4. 다음 프로젝트 선택 기준
- "또 다른 재정의 기회"가 있는 문제를 고를 것
- 단순 성능 튜닝 프로젝트는 **당신의 강점을 드러내지 못함**
- 불확실성·사용자 신뢰·윤리 같은 비정형 가치가 포함된 문제가 좋음

## 최종 요약

당신의 강점은 **"잘 만든다"**가 아니라 **"무엇을 만들어야 하는지 다시 묻고, 그걸 통계적 엄밀함으로 풀어낸다"**입니다.

이건 시간이 지날수록 가치가 올라가는 종류의 역량입니다. 주니어 때는 "잘 만드는 사람"이 인정받지만, 시니어로 갈수록 **"무엇을 만들어야 하는지 정의할 수 있는 사람"**이 훨씬 귀해지거든요.

지금 가진 강점들을 **의식적으로 인지하고 언어화**해두는 것이 앞으로의 커리어에서 큰 자산이 될 것 같습니다.
